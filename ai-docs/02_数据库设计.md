# 微信点餐小程序数据库设计文档

版本：V1.0
项目名称：微信点餐小程序
数据库方案：腾讯 CloudBase 云数据库
适用阶段：MVP 版本
协作方式：ChatGPT + Codex
文档日期：2026-06-06

---

## 1. 数据库设计目标

本数据库设计用于支持微信点餐小程序的第一阶段 MVP 开发，重点满足以下业务能力：

1. 用户浏览餐品。
2. 用户加入购物车并提交订单。
3. 商家接收订单。
4. 商家修改订单状态。
5. 商家维护餐品。
6. 商家维护分类。
7. 商家查看食材需求。
8. 商家查看商品制作流程。

本项目使用 CloudBase 云数据库，属于 NoSQL 文档型数据库。为了降低小白开发难度，数据库设计尽量保持清晰、稳定、少嵌套，便于 Codex 根据文档生成代码。

---

## 2. 设计原则

### 2.1 字段命名原则

所有字段统一使用小写字母加下划线命名，例如：

```text
merchant_id
created_at
updated_at
total_amount_cent
```

不要混用：

```text
merchantId
createdAt
totalAmount
```

---

### 2.2 金额字段原则

所有金额统一使用“分”为单位，字段名以 `_cent` 结尾。

示例：

```json
{
  "price_cent": 1800,
  "total_amount_cent": 5600
}
```

含义：

```text
1800 = 18.00 元
5600 = 56.00 元
```

禁止在数据库中直接保存浮点金额，例如：

```json
{
  "price": 18.8
}
```

原因：浮点数可能出现精度问题，订单金额必须准确。

---

### 2.3 时间字段原则

所有集合默认包含：

```text
created_at
updated_at
```

订单相关集合根据业务需要增加：

```text
accepted_at
cooking_at
finished_at
cancelled_at
paid_at
```

时间字段建议由云函数写入，使用服务端时间，避免完全相信前端时间。

---

### 2.4 权限原则

MVP 阶段建议核心数据读写都通过云函数完成。

前端不要直接修改以下数据：

```text
orders
order_items
dishes
categories
ingredients
dish_ingredients
production_steps
merchants
merchant_staff
```

前端可以请求云函数，由云函数完成权限校验和数据库操作。

---

### 2.5 订单快照原则

订单明细中必须保存用户下单时的餐品快照，包括：

```text
dish_name
dish_image_url
price_cent
quantity
subtotal_cent
```

这样即使商家后续修改餐品名称或价格，历史订单也不会受影响。

---

## 3. 数据库集合总览

MVP 阶段建议创建以下集合：

```text
users
merchants
merchant_staff
categories
dishes
ingredients
dish_ingredients
production_steps
orders
order_items
operation_logs
notifications
```

可选集合：

```text
carts
system_settings
```

说明：

1. `users`：用户信息。
2. `merchants`：商家信息。
3. `merchant_staff`：商家员工与权限关系。
4. `categories`：餐品分类。
5. `dishes`：餐品。
6. `ingredients`：食材。
7. `dish_ingredients`：餐品和食材的关系。
8. `production_steps`：餐品制作流程。
9. `orders`：订单主表。
10. `order_items`：订单明细表。
11. `operation_logs`：关键操作日志。
12. `notifications`：提醒记录。
13. `carts`：购物车，可选，MVP 阶段可优先用本地缓存。
14. `system_settings`：系统配置，可选。

### 8.4 status 可选值

```text
on_sale：上架中
off_sale：已下架
deleted：已删除 / 逻辑删除
```

注意：不再使用 `status = sold_out` 表示售罄。餐品 `status` 只负责上下架和逻辑删除，售罄由 `sold_out` 和库存规则判断。

---

### 8.5 库存 / 售罄规则

正式采用以下字段：

```js
{
  stock_enabled: false,
  stock_count: 0,
  sold_out: false
}
```

字段含义：

1. `stock_enabled`：是否启用库存管理，boolean，默认 `false`。
2. `stock_count`：剩余库存，number，必须是非负整数，默认 `0`。
3. `sold_out`：手动售罄开关，boolean，默认 `false`。

售罄判断统一使用：

```text
sold_out === true
或者
stock_enabled === true 且 stock_count <= 0
```

旧字段 `stock` 标记为废弃字段，仅用于兼容历史数据，不再作为后续新功能主字段。

---

### 8.6 规格 / 加料规则

P2-C 第一阶段规格 / 加料采用最小可用方案，直接嵌入 `dishes` 集合，不单独创建规格 / 加料集合。

第一阶段支持：

1. 规格支持单选。
2. 规格可配置必选。
3. 加料支持多选。
4. 加料可配置最多选择数量。
5. 规格和加料都支持加价。
6. 金额全部使用整数分，字段名以 `_cent` 结尾。

第一阶段不支持：

1. 不做 SKU 级库存。
2. 不做规格联动。
3. 不做每个规格独立图片。
4. 不做加料库存。
5. 不做负数加价。
6. 不做复杂优惠。
7. 不接支付。

默认值：

```js
{
  spec_groups: [],
  addon_groups: []
}
```

建议结构：

```js
{
  spec_groups: [
    {
      group_id: 'size',
      name: '规格',
      required: true,
      min_select: 1,
      max_select: 1,
      sort_order: 1,
      options: [
        {
          option_id: 'normal',
          name: '标准份',
          price_delta_cent: 0,
          enabled: true,
          sort_order: 1
        }
      ]
    }
  ],
  addon_groups: [
    {
      group_id: 'extra',
      name: '加料',
      required: false,
      min_select: 0,
      max_select: 3,
      sort_order: 1,
      options: [
        {
          option_id: 'egg',
          name: '加蛋',
          price_delta_cent: 200,
          enabled: true,
          sort_order: 1
        }
      ]
    }
  ]
}
```

字段校验规则：

1. `spec_groups` 必须是数组。
2. `addon_groups` 必须是数组。
3. `group_id` 不能为空。
4. 同一个餐品内 `group_id` 不能重复。
5. `group.name` 不能为空。
6. `required` 必须是 boolean。
7. `min_select` 必须是非负整数。
8. `max_select` 必须是非负整数。
9. `max_select` 不能小于 `min_select`。
10. `options` 必须是数组。
11. `option_id` 不能为空。
12. 同一个 group 内 `option_id` 不能重复。
13. `option.name` 不能为空。
14. `price_delta_cent` 必须是非负整数。
15. `enabled` 必须是 boolean。
16. `sort_order` 必须是数字。
17. 规格组第一阶段 `max_select` 固定为 `1`。
18. 加料组允许 `max_select` 大于 `1`。

  dish_id: 'dish_001',

## 18. 订单状态流转规则

### 18.1 状态值

```text
pending：待接单
accepted：已接单
cooking：制作中
finished：已完成
cancelled：已取消
```

---

### 18.2 允许的状态流转

```text
pending -> accepted
pending -> cancelled
accepted -> cooking
accepted -> cancelled
cooking -> finished
```

---

### 18.3 禁止的状态流转

```text
finished -> pending
finished -> accepted
finished -> cooking
finished -> cancelled

cancelled -> pending
cancelled -> accepted
cancelled -> cooking
cancelled -> finished

pending -> cooking
pending -> finished
accepted -> finished
cooking -> accepted
```

---

### 18.4 状态修改规则

1. 用户只能取消 `pending` 状态的订单。
2. 商家可以将 `pending` 改为 `accepted`。
3. 商家可以将 `accepted` 改为 `cooking`。
4. 商家可以将 `cooking` 改为 `finished`。
5. `finished` 和 `cancelled` 是终态，不允许继续修改。
6. 所有状态修改必须在云函数中校验。
7. 不允许前端直接修改订单状态。

## 20. 常用查询场景

### 20.1 用户端获取菜单

查询逻辑：

1. 根据 `merchant_id` 查询启用分类。
2. 根据 `merchant_id` 查询 `status = on_sale` 的餐品。
3. 前端按 `category_id` 分组展示。

涉及集合：

```text
categories
dishes
```

---

### 20.2 用户提交订单

处理逻辑：

1. 前端传入购物车餐品 ID、数量、已选规格 ID 和已选加料 ID。
2. 前端可以传 `selected_specs`，但只能作为规格选择 ID。
3. 前端可以传 `selected_addons`，但只能作为加料选择 ID。
4. 规格 / 加料名称、启用状态、加价必须由 `createOrder` 从数据库重新读取。
5. 云函数重新查询 `dishes`。
6. 云函数校验餐品是否上架。
7. 云函数校验库存 / 售罄。
8. 云函数校验必选规格必须已选择。
9. 云函数校验规格选项必须存在且 `enabled = true`。
10. 云函数校验加料选项必须存在且 `enabled = true`。
11. 云函数校验加料选择数量必须满足 `min_select` / `max_select`。
12. 规格 / 加料价格不能信任前端。
13. 云函数重新计算金额：

```text
unit_price_cent = price_cent + spec_delta_cent + addon_delta_cent
subtotal_cent = unit_price_cent * quantity
```

14. 写入 `orders`。
15. 写入 `order_items`。
16. 可选：写入 `notifications`。
17. 可选：写入 `operation_logs`。

涉及集合：

```text
dishes
orders
order_items
notifications
operation_logs
```

---

### 20.3 用户查看订单列表

查询逻辑：

1. 根据当前用户 `openid` 查询 `orders`。
2. 按 `created_at` 倒序排列。
3. 分页返回。

涉及集合：

```text
orders
```

---

### 20.4 用户查看订单详情

查询逻辑：

1. 根据 `order_id` 查询 `orders`。
2. 校验 `orders.user_openid` 是否等于当前用户 openid。
3. 查询对应 `order_items`。
4. 返回订单详情。

涉及集合：

```text
orders
order_items
```

---

### 20.5 商家查看订单列表

查询逻辑：

1. 根据当前员工 openid 查询 `merchant_staff`。
2. 获得 `merchant_id`。
3. 根据 `merchant_id` 查询 `orders`。
4. 可按 `status` 筛选。
5. 可按日期筛选。
6. 按 `created_at` 倒序排列。

涉及集合：

```text
merchant_staff
orders
```

---

### 20.6 商家修改订单状态

处理逻辑：

1. 根据当前员工 openid 查询 `merchant_staff`。
2. 校验员工是否属于该订单的商家。
3. 查询订单当前状态。
4. 校验状态流转是否合法。
5. 更新订单状态和对应时间字段。
6. 写入 `operation_logs`。
7. 可选：写入 `notifications`。

涉及集合：

```text
merchant_staff
orders
operation_logs
notifications
```

---

### 20.7 商家查看食材汇总

处理逻辑：

1. 商家选择日期范围。
2. 查询该时间范围内的订单。
3. 查询订单明细。
4. 根据 `dish_id` 查询 `dish_ingredients`。
5. 按 `ingredient_id` 汇总数量。
6. 返回食材名称、单位、总数量。

涉及集合：

```text
orders
order_items
dish_ingredients
ingredients
```

## 21. 权限设计

### 21.1 普通用户权限

普通用户允许：

1. 查看已上架餐品。
2. 创建自己的订单。
3. 查看自己的订单。
4. 取消自己的待接单订单。

普通用户禁止：

1. 查看其他用户订单。
2. 修改餐品。
3. 修改分类。
4. 修改订单状态为商家状态。
5. 查看商家后台数据。

---

### 21.2 商家用户权限

商家用户允许：

1. 查看自己商家的订单。
2. 修改自己商家的订单状态。
3. 管理自己商家的餐品。
4. 管理自己商家的分类。
5. 管理自己商家的食材。
6. 管理自己商家的制作流程。

商家用户禁止：

1. 查看其他商家的订单。
2. 修改其他商家的餐品。
3. 修改其他商家的分类。
4. 修改其他商家的员工。
5. 删除历史订单。

---

### 21.3 管理员权限

管理员允许：

1. 查看所有商家。
2. 查看所有订单。
3. 处理异常数据。
4. 管理商家状态。
5. 管理用户角色。

MVP 阶段可以暂不开发完整管理员后台。

## 24. 数据校验规则

### 24.1 餐品校验

新增或编辑餐品时必须校验：

1. `name` 不能为空。
2. `merchant_id` 不能为空。
3. `category_id` 不能为空。
4. `price_cent` 必须大于 0。
5. `stock_enabled` 必须是 boolean。
6. `stock_count` 必须是非负整数。
7. `sold_out` 必须是 boolean。
8. `status` 必须是合法状态。
9. 上架餐品建议必须有图片。

旧字段 `stock` 为废弃字段，不再作为新增或编辑餐品时的主库存字段。

---

### 24.2 分类校验

新增或编辑分类时必须校验：

1. `name` 不能为空。
2. `merchant_id` 不能为空。
3. `sort_order` 必须是数字。
4. `enabled` 必须是布尔值。

---

### 24.3 订单校验

创建订单时必须校验：

1. 购物车不能为空。
2. 商品数量必须大于 0。
3. 餐品必须存在。
4. 餐品必须为 `on_sale`。
5. 库存必须足够。
6. 前端可以传 `selected_specs` / `selected_addons`，但只能作为选择 ID。
7. 必选规格必须已选择。
8. 规格选项必须存在。
9. 规格选项必须 `enabled = true`。
10. 加料选项必须存在。
11. 加料选项必须 `enabled = true`。
12. 加料选择数量必须满足 `min_select` / `max_select`。
13. 规格 / 加料价格不能信任前端。
14. `unit_price_cent` 必须由云函数重新计算。
15. `subtotal_cent` 必须由云函数重新计算。
16. `total_amount_cent` 必须由云函数重新计算。
17. 订单金额必须由云函数重新计算。
18. 用户 openid 必须从云函数上下文获取。
19. 订单编号必须唯一。

金额计算公式：

```text
unit_price_cent = price_cent + spec_delta_cent + addon_delta_cent
subtotal_cent = unit_price_cent * quantity
```

---

### 24.4 订单状态校验

修改订单状态时必须校验：

1. 当前用户必须是商家员工。
2. 订单必须属于当前商家。
3. 当前状态必须允许流转到目标状态。
4. `finished` 和 `cancelled` 不允许再次修改。
5. 状态时间字段必须由云函数写入。

---

## 25. 订单编号规则

订单编号建议格式：

```text
YYYYMMDD + 4位递增数字
```

示例：

```text
202606060001
202606060002
202606060003
```

MVP 阶段也可以使用：

```text
YYYYMMDDHHmmss + 3位随机数
```

示例：

```text
20260606153212085
```

推荐 MVP 先使用第二种，因为实现更简单，不需要处理同一天递增序号并发问题。

## 26. 数据库变更规则

为了避免 Codex 随意修改数据库结构，后续所有字段变更必须遵守以下规则：

1. 新增字段必须先更新本文档。
2. 删除字段必须确认没有页面和云函数使用。
3. 修改字段名必须同步修改所有相关代码。
4. 订单相关字段不能随意删除。
5. 金额字段必须保持 `_cent` 规则。
6. 状态字段必须保持本文档定义的枚举值。
7. 每次数据库变更必须记录到迭代文档。

---

## 27. Codex 开发约束

Codex 在开发数据库相关功能时必须遵守：

1. 不允许临时新增未在本文档定义的集合。
2. 不允许随意修改字段命名。
3. 不允许在前端直接修改订单状态。
4. 不允许前端直接信任订单金额。
5. 不允许删除历史订单。
6. 不允许绕过 `merchant_staff` 做商家权限判断。
7. 所有商家数据查询必须带上 `merchant_id`。
8. 所有用户订单查询必须带上当前用户 `openid`。
9. 所有金额字段必须使用分为单位。
10. 所有状态字段必须使用本文档定义的枚举值。
