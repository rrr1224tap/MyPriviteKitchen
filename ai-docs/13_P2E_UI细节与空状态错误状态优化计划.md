# P2-E UI细节与空状态错误状态优化计划

文档日期：2026-06-22

阶段目标：在不新增业务能力的前提下，收口用户端和商家端的加载、空状态、错误状态、按钮禁用、异常提示和视觉一致性，让已经完成的点餐、购物车、下单、订单查看、商家处理订单链路更稳定、更适合演示和体验版测试。

本次文档只做影响面检查和计划整理，不修改小程序页面、不修改云函数、不修改业务代码。

---

## 一、阶段目标

P2-E 不是新功能阶段，而是体验收口阶段。

本阶段重点目标：

1. 主链路中所有失败场景都有用户可理解的提示。
2. 关键按钮在请求中有禁用或防重复点击处理。
3. 空状态、错误状态、加载状态在用户端和商家端保持一致。
4. 售罄、库存不足、规格 / 加料错误等提示对普通用户友好。
5. 页面继续保持当前玻璃风 / 暖色商业餐饮风格，不做大规模 UI 重做。
6. 旧购物车、旧订单、旧餐品数据仍然兼容，不因为缺字段导致页面崩溃。

## 三、当前页面现状总结

### 1. 用户菜单页

现状：

1. 有 `loading` 状态。
2. 有空菜单状态。
3. 云函数失败时会使用 fallback 菜单继续展示。
4. 已支持售罄展示和禁用加购。
5. 已支持有规格餐品显示“选规格”。
6. 图片加载失败有兜底图。

需要关注：

1. 云函数失败使用 fallback 后，用户可能不知道当前不是线上真实菜单。
2. 搜索入口目前只提示“后续接入”，不是真搜索。
3. 无搜索结果状态目前不存在，因为搜索功能未接入。
4. 菜单接口失败和真实空菜单的状态区分还可以更清晰。

### 2. 用户餐品详情页

现状：

1. 有 `loading` 文案。
2. 缺少 `dish_id` 或云函数失败时会展示 fallback 餐品。
3. 已支持规格 / 加料选择。
4. 已支持售罄时禁用加入购物车。
5. 已支持加料超过 `max_select` 时提示。
6. 图片加载失败有兜底图。

需要关注：

1. 缺少 `dish_id` 时展示 fallback 餐品，可能让用户误以为可以下单。
2. 云函数失败展示 fallback 餐品，适合开发兜底，但上线前需要改成更明确的错误状态或重试入口。
3. 加入购物车按钮只在 `loading` 时禁用，售罄状态依赖样式和逻辑拦截，禁用表达可以更清晰。
4. 必选规格未选、加料不足等提示目前存在，但文案可进一步统一。

### 3. 用户购物车页

现状：

1. 购物车为空时有完整空状态和“去点餐”入口。
2. 提交订单时有 `submitting` 状态，按钮禁用并显示提交中。
3. 已支持规格 / 加料快照展示。
4. 已支持 `item_key`，避免不同规格 / 加料混合。
5. 图片加载失败有兜底。
6. 下单成功有弹窗，可查看订单或继续点餐。

需要关注：

1. `createOrder` 未知错误码时，当前依赖云函数调用封装的默认 toast，购物车自身没有统一兜底文案。
2. `DISH_SOLD_OUT`、`STOCK_NOT_ENOUGH` 等已映射，但可以在失败后引导用户刷新购物车或回菜单重新选择。
3. 清空购物车有确认弹窗，但减少数量到 0 时直接移除，没有二次确认；目前不阻塞主链路。
4. 底部提交栏在极小屏幕、键盘弹起或安全区场景下仍需要真机检查。

### 4. 用户订单列表页

现状：

## 四、用户端问题清单

### P0：影响主链路或可能导致误操作

1. 餐品详情页缺少 `dish_id` 或加载失败时展示 fallback 餐品，可能让用户误以为是真实餐品并加入购物车。
2. 购物车提交订单遇到未知错误码时，页面自身缺少兜底错误映射，用户可能只看到较技术化或不完整提示。
3. 购物车中旧缓存餐品如果已下架、售罄或规格失效，最终由 `createOrder` 拦截，但页面没有引导用户移除问题商品。

### P1：明显影响体验但不阻塞主链路

1. 菜单页云函数失败后直接使用 fallback 菜单，缺少“当前为示例菜单 / 重试”的明确提示。
2. 用户订单列表和订单详情错误文案可能直接显示云函数 message，需要统一转成普通用户可理解文案。
3. 订单状态文案在多个页面中重复定义，后续维护时可能不一致。
4. 售罄、库存不足、规格失效后没有统一引导用户回到菜单或刷新。
5. 长菜名、长规格 / 加料摘要在小屏幕上可能挤压价格和按钮区域。
6. 搜索入口目前未实现，只显示后续接入提示，体验上可以弱化或标记为开发中。

### P2：纯视觉打磨

1. 空状态 icon / 标识在不同页面风格略有差异，可统一为一套餐饮风 icon。
2. 加载状态多为文字和省略号，可以统一为更有品牌感的轻量 loading。
3. 状态 badge 的颜色和字号可以进一步统一。
4. 图片兜底样式可统一，减少不同页面之间的视觉差异。
5. 价格、数量、规格 / 加料文字层级可以进一步统一。

## 五、商家端问题清单

5. 分类停用和餐品下架的确认文案可以补充“不会影响历史订单”。

### P2：纯视觉打磨

1. 商家端和用户端的 state-card 可以统一间距、字号和 icon。

3. 商家工作台入口卡片可以在后续加入轻量状态数字，但本阶段不建议开发统计功能。

## 六、空状态优化清单

建议补齐或优化：

1. 菜单无分类：提示“菜单还没准备好”，引导稍后再试。
2. 菜单有分类但无餐品：提示“当前分类暂无可点餐品”，保留分类切换。
3. 搜索无结果：搜索功能接入后再补。
4. 购物车为空：当前已有，可保留。
5. 用户订单列表为空：当前已有，可保留并统一文案。
6. 用户订单详情找不到：当前已有，可保留。
7. 商家订单列表为空：区分“全部暂无订单”和“当前筛选暂无订单”。
8. 商家订单详情找不到：当前已有，可保留。
9. 商家分类为空：保留“新增分类”入口。
10. 商家餐品为空：保留“新增餐品”入口，并提示需要先有启用分类。

## 七、错误状态优化清单

建议按错误来源统一处理：

1. 网络失败：提示“网络不太稳定，请稍后重试”。
2. 云函数失败：提示“服务暂时不可用，请稍后重试”。
3. 商家权限失败 `FORBIDDEN`：提示“当前账号没有商家权限，请先配置 merchant_staff”。
4. 餐品下架 `DISH_OFF_SALE`：提示“有餐品已下架，请重新选择”。
5. 餐品售罄 `DISH_SOLD_OUT`：提示“有餐品已售罄，请重新选择”。
6. 库存不足 `STOCK_NOT_ENOUGH`：提示“库存不足，请调整数量”。
7. 规格 / 加料无效 `VALIDATION_ERROR`：提示“规格或加料选择已变化，请重新选择”。
8. 参数错误 `INVALID_PARAMS`：提示“订单信息不完整，请重新选择”。
9. 数据异常：提示“页面数据异常，请刷新重试”。
10. 未知错误：必须有兜底提示，不展示 `DATABASE_ERROR` 等技术字段。

## 八、加载状态 / 防重复点击优化清单

建议检查并优化：

1. 菜单加载时有 loading，但云函数失败 fallback 应有提示和重试入口。
2. 详情页加载失败不建议继续展示可下单 fallback。
3. 购物车提交订单已有 `submitting`，继续保留。
4. 商家订单列表操作已有 `submittingOrderId`，建议增加确认弹窗。
5. 商家订单详情操作已有 `submitting`，建议增加确认弹窗。
6. 商家餐品保存、图片上传已有禁用，继续保留。
7. 商家分类操作已有 `submitting`，继续保留。
8. 所有重试按钮在请求中可以禁用，避免连续点击。

## 九、文案与视觉细节优化清单

文案建议：

1. 不向用户展示 `group_id`、`option_id`、`DATABASE_ERROR` 等技术字段。
2. 用户端使用“餐品已售罄 / 库存不足 / 请重新选择”这类直白文案。
3. 商家端使用“当前账号没有商家权限 / 订单状态已变化 / 请刷新后重试”这类可执行文案。
4. 订单状态文案统一从常量读取，减少重复硬编码。

视觉建议：

1. 保持玻璃风 / 暖色商业餐饮风。
2. 不使用系统默认蓝色按钮。
3. 空状态使用统一的圆形 icon + 标题 + 说明 + 主按钮结构。
4. 错误状态使用统一的感叹号 icon + 重试按钮。
5. 价格颜色保持品牌红，数量和辅助信息降低权重。
6. 长菜名、长规格 / 加料摘要建议最多两行，超出截断。
7. 底部固定按钮需要保留安全区 padding。
8. 图片加载失败统一使用项目已有 fallback 图片或暖色占位。

## 十、优先级分组

### P0：影响主链路或会导致误操作

1. 餐品详情页加载失败 / 缺少 `dish_id` 时不应继续展示可下单 fallback。

### P2：纯视觉打磨问题

## 十一、建议拆分任务

### P2-E-1：用户端空状态 / 错误状态基础收口

目标：

1. 收口菜单页、餐品详情页、订单列表页、订单详情页的错误和空状态。
2. 重点修复详情页 fallback 可下单风险。
3. 菜单页 fallback 增加明确提示和重试入口。

修改范围：

1. `miniprogram/pages/user/menu/`
2. `miniprogram/pages/user/dish-detail/`
3. `miniprogram/pages/user/order-list/`
4. `miniprogram/pages/user/order-detail/`

不修改范围：

1. 不修改云函数。
2. 不修改购物车提交逻辑。
3. 不修改商家端页面。

测试方式：

1. 模拟菜单云函数失败，确认有提示和重试。
2. 直接打开无 `dish_id` 的详情页，确认显示错误而不是可下单餐品。
3. 断网或模拟订单接口失败，确认订单页错误状态可重试。

### P2-E-2：购物车 / 下单防重复与错误提示优化

目标：

1. 完善 `createOrder` 错误码到用户友好文案的映射。
2. 未知错误也必须有兜底提示。
3. 售罄、库存不足、规格失效后给用户明确处理建议。

修改范围：

1. `miniprogram/pages/user/cart/`

不修改范围：

1. 不修改 `createOrder`。
2. 不修改详情页和菜单页。
3. 不接支付。

测试方式：

1. 购物车为空点击提交。
2. 模拟 `DISH_SOLD_OUT`。
3. 模拟 `STOCK_NOT_ENOUGH`。
