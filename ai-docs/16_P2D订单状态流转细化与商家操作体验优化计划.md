# P2-D 订单状态流转细化与商家操作体验优化计划

文档日期：2026-06-22

当前稳定基线：`prelaunch-check-v0.1`

基线提交：`d8af771cdc15b6b246564c1ab712d20115f10333`

---

## 一、阶段目标

P2-D 的目标是把订单状态流转、商家操作反馈、用户端状态感知和异常状态处理进一步稳定化，为后续 P3-A 支付前准备打基础。

本阶段重点：

1. 不接微信支付。
2. 不做退款。
3. 不做打印机。
4. 不做优惠券。
5. 不做多门店。
6. 不改变当前主链路。
7. 只围绕订单状态流转和操作体验继续细化。

P2-D 不追求新增复杂商业能力，优先保证订单状态更稳、操作反馈更清楚、非法流转更难发生。

---

## 二、当前订单状态基线

当前订单状态以项目真实状态为准：

```text
pending：待接单
accepted：已接单 / 待制作
cooking：制作中
finished：已完成
cancelled：已取消
```

当前已实现：

1. `createOrder` 创建订单后为 `pending`。
2. 商家可从 `pending` 接单到 `accepted`。
3. 商家可从 `accepted` 开始制作到 `cooking`。
4. 商家可从 `cooking` 完成到 `finished`。
5. 用户端订单列表 / 详情可展示状态。
6. 商家端订单列表 / 详情可展示和操作状态。
7. 商家操作已有二次确认。
8. 操作中已有防重复提交。
9. `cancelled` 目前主要是兼容展示，取消入口需要单独评估。

---

## 三、目标状态流转图

建议状态流转：

```text
pending -> accepted -> cooking -> finished
pending -> cancelled
accepted -> cancelled
```

可选讨论：

```text
cooking -> cancelled 是否允许：默认不建议在本阶段开放，避免商家已制作后误取消。
finished -> cancelled 是否允许：不允许。
cancelled -> 其他状态：不允许。
```

注意：

1. 当前未接支付，所以取消不涉及退款。
2. 后续接支付前，需要重新设计 `paid` / `refunding` / `refunded` 等支付相关状态。
3. P2-D 不引入支付状态。

---

## 四、P2-D 拆分建议

### P2-D-1：后端订单状态流转守卫检查

目标：

1. 检查 `updateOrderStatus` 是否严格限制合法状态流转。
2. 防止非法跳转：
   - `pending -> cooking`
   - `pending -> finished`
   - `accepted -> finished`
   - `cooking -> accepted`
   - `finished -> 任意状态`
   - `cancelled -> 任意状态`
3. 错误码统一：
   - `INVALID_STATUS`
   - `STATUS_CONFLICT`
   - `ORDER_NOT_FOUND`
   - `FORBIDDEN`
4. 不改变订单主结构。

建议修改范围：

1. `cloudfunctions/updateOrderStatus/`
2. 对应测试文件，如当前已有。

不修改范围：

1. 不修改用户端页面。
2. 不修改商家端页面。
3. 不修改订单数据结构。

### P2-D-2：商家端取消订单能力评估与实现

目标：

1. 评估是否增加商家取消订单按钮。
2. 建议只允许：
   - `pending -> cancelled`
   - `accepted -> cancelled`
3. `cooking` / `finished` 不允许取消。
4. 取消前必须二次确认。
5. 文案明确：取消后状态不可回退。
6. 不涉及退款。

建议修改范围：

1. `updateOrderStatus` 如后端还未支持取消流转。
2. `miniprogram/pages/merchant/orders/`
3. `miniprogram/pages/merchant/order-detail/`

不修改范围：

1. 不接支付。
2. 不做退款。
3. 不修改用户端取消入口。

### P2-D-3：用户端取消订单能力评估与实现

目标：

1. 评估是否允许用户取消订单。
2. 建议只允许：
   - `pending -> cancelled`
3. `accepted` / `cooking` / `finished` 不允许用户取消。
4. 取消前必须二次确认。
5. 不涉及退款。
6. 后续接支付后重新设计退款流程。

建议修改范围：

1. 如需要，可新增或扩展用户取消订单云函数能力。
2. `miniprogram/pages/user/order-list/`
3. `miniprogram/pages/user/order-detail/`

不修改范围：

1. 不接支付。
2. 不做退款。
3. 不改变商家已有状态操作。

### P2-D-4：用户端订单状态刷新体验优化

目标：

1. 用户订单详情增加更明确的刷新入口。
2. 用户订单列表下拉刷新状态稳定。
3. 状态变更后用户端文案更明确。
4. 不做实时 WebSocket / 订阅消息。
5. 不接消息推送。

建议修改范围：

1. `miniprogram/pages/user/order-list/`
2. `miniprogram/pages/user/order-detail/`

不修改范围：

1. 不修改云函数状态流转规则。
2. 不新增推送能力。
3. 不做实时通信。

### P2-D-5：商家端操作结果反馈优化

目标：

1. 操作成功 toast 文案统一。
2. 操作失败错误码转译保持一致。
3. 操作后列表 / 详情刷新稳定。
4. 防止重复点击。
5. 操作按钮在非法状态不出现。

建议修改范围：

1. `miniprogram/pages/merchant/orders/`
2. `miniprogram/pages/merchant/order-detail/`

不修改范围：

1. 不修改订单状态流转规则。
2. 不接支付。
3. 不做统计看板。

### P2-D-6：P2-D 联调与验收文档

目标：

1. 汇总 P2-D 状态流转。
2. 列出自动测试。
3. 列出手动验收清单。
4. 决定是否打标签：

```text
p2-order-status-v0.1
```

## 五、后端安全原则

1. 前端按钮隐藏不是安全边界。
2. 所有状态流转必须以后端校验为准。
3. 商家身份必须后端校验。
4. `openid` 不可信，必须从云函数上下文获取。
5. 订单不存在、状态冲突、非法状态都要有明确错误码。
6. `finished` / `cancelled` 终态不允许再流转。
7. 后续接支付前，不能把取消和退款混在一起。
8. 云函数返回格式继续保持统一 `success / code / message / data`。

---

## 六、用户体验原则

1. 所有不可逆操作必须二次确认。
2. 操作中按钮禁用。
3. 操作失败不改变本地状态。
4. 操作成功后刷新远端数据。
5. 用户端和商家端状态文案保持一致。
6. 不向用户展示技术错误字段。
7. `cancelled` 状态要有明确视觉区分。
8. `finished` 状态要有明确完成反馈。
9. 取消能力如果上线，必须明确告知“取消后不可恢复”。
10. 未接支付前，取消文案不得暗示退款已处理。

---

## 七、验收重点

- [ ] `pending` 订单可被商家接单。
- [ ] `accepted` 订单可开始制作。
- [ ] `cooking` 订单可完成。
- [ ] `finished` 订单不能再操作。
- [ ] `cancelled` 订单不能再操作。
- [ ] 非法状态跳转被后端拒绝。
- [ ] 重复点击不会重复更新。
- [ ] 商家无权限不能更新订单。
- [ ] 用户端订单状态展示正确。
- [ ] 商家端订单状态展示正确。
- [ ] 取消订单能力如果实现，只允许规定状态取消。
- [ ] 所有取消 / 完成等不可逆操作都有确认弹窗。
- [ ] 不涉及支付 / 退款。

---

## 八、阶段边界

P2-D 不做：

1. 微信支付。
2. 退款。
3. 订阅消息。
4. 打印机。
5. 后厨看板。
6. 多门店。
7. 订单统计。
8. 配送。
9. 复杂售后。
10. 实时推送。

---

推荐顺序：

2. P2-D-2 商家端取消订单能力。
3. P2-D-3 用户端取消订单能力。
4. P2-D-4 用户端状态刷新体验。
5. P2-D-5 商家端操作反馈。
6. P2-D-6 联调与验收。
7. 打标签：
