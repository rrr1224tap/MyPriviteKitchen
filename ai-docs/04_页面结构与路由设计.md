# 微信点餐小程序页面结构与路由设计文档

版本：V1.0
项目名称：微信点餐小程序
技术方案：微信原生小程序 + 腾讯 CloudBase 云开发
适用阶段：MVP 版本
协作方式：ChatGPT + Codex
文档日期：2026-06-06

---

## 1. 文档目标

本文档用于规范微信点餐小程序的页面结构、页面路径、页面功能、页面跳转关系和页面调用的云函数。

本项目包含两个主要端：

1. 用户端：顾客浏览餐品、加入购物车、提交订单、查看订单状态。
2. 商家端：商家查看订单、处理订单、维护餐品、维护分类、查看食材汇总、管理制作流程。

本文档的核心目标是让 Codex 在开发页面时：

1. 不随意创建页面路径。
2. 不随意修改页面命名。
3. 不混乱用户端和商家端页面。
4. 每个页面只负责自己的功能。
5. 页面调用的云函数和数据结构保持一致。
6. 后续迭代时能够按页面逐步扩展。

---

## 2. 页面设计原则

### 2.1 用户端和商家端分离

用户端页面统一放在：

```text
miniprogram/pages/user/
```

商家端页面统一放在：

```text
miniprogram/pages/merchant/
```

通用页面统一放在：

```text
miniprogram/pages/common/
```

通用组件统一放在：

```text
miniprogram/components/
```

通用工具函数统一放在：

```text
miniprogram/utils/
```

---

### 2.2 页面命名原则

页面路径统一使用小写字母和短横线命名。

推荐：

```text
menu
dish-detail
cart
order-list
order-detail
```

不推荐：

```text
dishDetail
OrderList
order_detail
```

---

### 2.3 页面职责原则

每个页面只负责一个核心目标。

例如：

1. `menu` 只负责展示菜单和加入购物车。
2. `cart` 只负责购物车确认和提交订单入口。
3. `order-detail` 只负责展示订单详情和状态。
4. `merchant/orders` 只负责商家订单列表。
5. `merchant/dish-edit` 只负责新增或编辑餐品。

不要把多个复杂功能堆到一个页面中。

---

### 2.4 云函数调用原则

页面不直接操作核心数据库。

页面通过云函数完成：

1. 登录。
2. 获取菜单。
3. 创建订单。
4. 查询订单。
5. 修改订单状态。
6. 新增或编辑餐品。
7. 新增或编辑分类。
8. 食材汇总。
9. 制作流程管理。

---

### 2.5 路由跳转原则

用户端页面之间使用普通跳转。

商家端页面需要先校验当前用户是否为商家员工。

如果用户不是商家员工，不允许进入商家端页面。

## 6. 通用页面设计

## 6.1 启动页

### 页面路径

```text
pages/common/launch/launch
```

### 页面目标

作为小程序启动入口，完成初始化和登录判断。

### 页面职责

1. 初始化云开发环境。
2. 调用 `login` 云函数。
3. 保存用户信息到全局状态或本地缓存。
4. 判断用户身份。
5. 默认跳转到用户菜单页。
6. 如果用户是商家员工，可提供进入商家端入口。

### 调用云函数

```text
login
```

### 页面数据

```js
{
  loading: true,
  userInfo: null,
  merchantStaff: null,
  merchant: null
}
```

### 页面跳转

普通用户：

```text
launch -> user/menu
```

商家用户：

```text
launch -> user/menu
```

商家用户也可以后续通过个人中心或入口进入：

```text
user/profile -> merchant/dashboard
```

### MVP 建议

MVP 阶段不要做复杂角色选择。启动后默认进入用户端菜单页，如果当前用户有商家权限，在页面上显示“进入商家端”按钮。

---

## 6.2 角色选择页，可选

### 页面路径

```text
pages/common/role-select/role-select
```

### 页面目标

当用户同时拥有普通用户和商家身份时，选择进入用户端还是商家端。

### 页面职责

1. 展示“我要点餐”入口。
2. 展示“商家管理”入口。
3. 根据登录信息判断是否显示商家入口。

### 页面跳转

```text
role-select -> user/menu
role-select -> merchant/dashboard
```

### MVP 建议

此页面可暂缓开发。

## 7. 用户端页面设计

## 7.1 用户菜单页

### 页面路径

```text
pages/user/menu/menu
```

### 页面目标

让用户浏览餐品、按分类筛选、加入购物车。

### 页面内容

1. 店铺名称。
2. 店铺公告。
3. 营业状态。
4. 餐品分类栏。
5. 餐品列表。
6. 餐品图片。
7. 餐品名称。
8. 餐品价格。
9. 餐品描述。
10. 加入购物车按钮。
11. 购物车悬浮栏。
12. 进入订单列表入口。
13. 如果当前用户是商家员工，显示“商家端”入口。

### 调用云函数

```text
getMenu
```

### 页面入参

```text
merchant_id
```

MVP 阶段可以使用默认商家 ID：

```text
merchant_001
```

后续多商家版本再改为动态商家 ID。

### 页面数据

```js
{
  merchantId: 'merchant_001',
  merchant: null,
  categories: [],
  dishes: [],
  currentCategoryId: '',
  cartItems: [],
  cartTotalCount: 0,
  cartTotalAmountCent: 0,
  loading: false
}
```

### 核心交互

1. 页面加载时调用 `getMenu`。
2. 点击分类后切换当前分类。
3. 点击餐品卡片进入餐品详情页。
4. 点击加号加入购物车。
5. 点击购物车栏进入购物车页。
6. 点击订单入口进入用户订单列表页。
7. 点击商家端入口进入商家首页。

### 跳转关系

```text
menu -> dish-detail
menu -> cart
menu -> order-list
menu -> merchant/dashboard
```

### 验收标准

1. 页面能展示商家名称和公告。
2. 页面能展示分类。
3. 页面能展示已上架餐品。
4. 下架餐品不展示。
5. 点击分类后餐品筛选正确。
6. 点击加入购物车后数量增加。
7. 购物车总数量和总金额正确。
8. 点击购物车可以进入购物车页。

---

## 7.2 餐品详情页

### 页面路径

```text
pages/user/dish-detail/dish-detail
```

### 页面目标

展示单个餐品详情，并支持加入购物车。

### 页面内容

1. 餐品主图。
2. 餐品名称。
3. 餐品价格。
4. 餐品描述。
5. 标签。
6. 预计制作时间。
7. 库存或售罄状态。
8. 数量选择。
9. 加入购物车按钮。
10. 返回菜单按钮。

### 调用云函数

```text
getDishDetail
```

### 页面入参

```text
dish_id
```

示例：

```text
/pages/user/dish-detail/dish-detail?dish_id=dish_001
```

### 页面数据

```js
{
  dishId: '',
  dish: null,
  quantity: 1,
  loading: false
}
```

### 核心交互

1. 页面加载时读取 `dish_id`。
2. 调用 `getDishDetail` 获取餐品详情。
3. 用户可以增加或减少数量。
4. 点击加入购物车后写入本地购物车。
5. 加入成功后提示用户。

### 跳转关系

```text
dish-detail -> cart
dish-detail -> menu
```

### 验收标准

1. 能正确展示餐品详情。
2. 无 `dish_id` 时提示错误。
3. 餐品不存在时提示错误。
4. 已下架餐品不可加入购物车。
5. 已售罄餐品不可加入购物车。
6. 数量不能小于 1。
7. 加入购物车后数据正确。

## 7.3 购物车页

### 页面路径

```text
pages/user/cart/cart
```

### 页面目标

让用户查看已选餐品、修改数量、确认下单。

### 页面内容

1. 已选餐品列表。
2. 餐品图片。
3. 餐品名称。
4. 单价。
5. 数量加减。
6. 小计。
7. 清空购物车按钮。
8. 总数量。
9. 总金额。
10. 去结算按钮。

### 调用云函数

MVP 阶段购物车使用本地缓存，不需要直接调用云函数。

提交订单在 `submit-order` 页面调用：

```text
createOrder
```

### 页面数据

```js
{
  cartItems: [],
  totalCount: 0,
  totalAmountCent: 0,
  isEmpty: true
}
```

### 核心交互

1. 页面显示本地购物车。
2. 用户可以增加数量。
3. 用户可以减少数量。
4. 数量为 0 时移除该餐品。
5. 用户可以清空购物车。
6. 点击去结算进入提交订单页。

### 跳转关系

```text
cart -> submit-order
cart -> menu
```

### 验收标准

1. 能正确读取本地购物车。
2. 商品数量修改正确。
3. 金额计算正确。
4. 清空购物车有效。
5. 购物车为空时不能结算。
6. 点击去结算能进入提交订单页。

## 7.4 提交订单页

### 页面路径

```text
pages/user/submit-order/submit-order
```

### 页面目标

让用户确认订单信息并提交订单。

### 页面内容

1. 订单餐品明细。
2. 总数量。
3. 总金额。
4. 取餐方式。
5. 联系电话。
6. 用户备注。
7. 提交订单按钮。

### 调用云函数

```text
createOrder
```

### 页面数据

```js
{
  merchantId: 'merchant_001',
  cartItems: [],
  totalCount: 0,
  totalAmountCent: 0,
  pickupType: 'pickup',
  contactPhone: '',
  remark: '',
  submitting: false
}
```

### 提交给云函数的数据

```js
{
  merchant_id: 'merchant_001',
  items: [
    {
      dish_id: 'dish_001',
      quantity: 2
    }
  ],
  remark: '不要香菜',
  pickup_type: 'pickup',
  contact_phone: '13800000000'
}
```

注意：

页面可以展示前端计算的总金额，但真实订单金额以 `createOrder` 云函数计算结果为准。

### 核心交互

1. 页面加载购物车。
2. 用户填写联系电话。
3. 用户填写备注。
4. 用户点击提交订单。
5. 调用 `createOrder` 云函数。
6. 创建成功后清空购物车。
7. 跳转到订单详情页。

### 跳转关系

```text
submit-order -> order-detail
submit-order -> cart
```

### 验收标准

1. 购物车为空时不能提交订单。
2. 联系电话格式不合法时提示。
3. 提交过程中按钮显示 loading，防止重复提交。

## 17. 本地购物车设计

MVP 阶段购物车建议使用本地缓存，不创建购物车数据库集合。

### 17.1 本地缓存 key

```text
cart_items
```

### 17.2 数据结构

```js
[
  {
    "dish_id": "dish_001",
    "merchant_id": "merchant_001",
    "category_id": "category_001",
    "name": "招牌牛肉饭",
    "image": "",
    "price_cent": 2800,
    "quantity": 2,
    "stock": 99,
    "status": "on_sale"
  }
]
```

### 17.3 购物车规则

1. 同一餐品重复加入时，只增加数量。
2. 数量不能小于 1。
3. 数量为 0 时从购物车移除。
4. 购物车只允许同一商家的餐品。
5. 提交订单成功后清空购物车。
6. 下单时必须让云函数重新校验价格、库存和餐品状态。

```js
  hasMore: true
```

提交类页面建议包含：

```js

```

保存类页面建议包含：


## 20. 页面错误处理规范

3. 保留用户已输入内容。

```text
网络异常，请稍后重试
数据加载失败，请下拉刷新

你没有权限访问该页面
餐品已下架，无法下单
库存不足，请重新选择
```
