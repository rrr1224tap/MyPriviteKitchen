# P2-D 订单状态流转联调与验收清单

## 一、阶段结论

P2-D 订单状态流转细化与商家操作体验优化已完成。

本阶段已覆盖：

1. 后端订单状态流转守卫加固。
2. 商家端取消订单能力。
3. 用户端取消订单能力。
4. 用户端订单状态刷新体验优化。
5. 商家端订单操作结果反馈优化。

本阶段不接微信支付，不做退款，不新增复杂订单状态，不改变当前点餐主链路。P2-D 的完成为后续 P3-A 支付前准备打下基础。

## 二、最终订单状态机

当前订单状态集合：

```text
pending：待接单
accepted：已接单 / 待制作
cooking：制作中
finished：已完成
cancelled：已取消
```

合法状态流转：

```text
pending -> accepted
accepted -> cooking
cooking -> finished
pending -> cancelled
accepted -> cancelled
```

禁止状态流转：

```text
pending -> cooking
pending -> finished
accepted -> pending
accepted -> finished
cooking -> pending
cooking -> accepted
cooking -> cancelled
finished -> 任意状态
cancelled -> 任意状态
```

说明：

1. `finished` 和 `cancelled` 是终态，不允许继续流转。
2. 用户端只允许取消自己的 `pending` 订单。
3. 商家端允许取消 `pending` / `accepted` 订单。
4. `cooking` 订单不允许取消，避免已制作后误取消。
5. 当前未接支付，取消订单只代表订单状态取消，不涉及退款。

## 三、完成范围

### 3.1 后端 updateOrderStatus

1. 明确合法状态集合：`pending`、`accepted`、`cooking`、`finished`、`cancelled`。
2. 明确合法流转集合：`pending -> accepted`、`accepted -> cooking`、`cooking -> finished`、`pending -> cancelled`、`accepted -> cancelled`。
3. 拒绝非法流转。
4. 保护 `finished` / `cancelled` 终态。
5. 保持商家权限后端校验。
6. 保持统一错误码返回。
7. 状态流转测试 `25/25` 通过。

### 3.2 商家端取消订单

1. `pending` 订单可取消。
2. `accepted` 订单可取消。
3. `cooking` / `finished` / `cancelled` 不显示取消按钮。
4. 取消前必须二次确认。
5. 点击“再想想”不调用云函数。
6. 点击“确认取消”后调用 `updateOrderStatus`，目标状态为 `cancelled`。
7. 取消成功后刷新订单列表或订单详情。
8. 不涉及支付，不涉及退款。

### 3.3 用户端取消订单

1. 新增用户侧云函数 `cancelUserOrder`。
2. 用户只允许取消自己的 `pending` 订单。
3. 后端从 CloudBase 上下文获取 `openid`，不信任前端传入身份。
4. 后端校验订单归属当前用户。
5. 后端只允许 `pending -> cancelled`。
6. `accepted` / `cooking` / `finished` / `cancelled` 都不允许用户取消。
7. 用户订单列表和订单详情增加取消入口。
8. 取消前必须二次确认。
9. `cancelUserOrder` 测试 `12/12` 通过。
10. 不涉及支付，不涉及退款。

### 3.4 用户端状态刷新体验

1. 订单列表支持轻量刷新，有旧数据时刷新不先清空列表，减少闪烁。
2. 下拉刷新完成后调用 `wx.stopPullDownRefresh()`。
3. 订单列表显示最近更新时间。
4. 从订单详情返回订单列表时轻量刷新状态。
5. 订单详情增加“刷新状态”入口。
6. 刷新中禁用重复点击。
7. 用户取消订单成功后，列表 / 详情刷新为 `cancelled`。
8. `STATUS_CONFLICT` 使用友好文案提示，不展示技术字段。

### 3.5 商家端操作反馈

1. 操作成功提示统一：

```text
pending -> accepted：已接单
accepted -> cooking：已开始制作
cooking -> finished：订单已完成
pending / accepted -> cancelled：订单已取消
```

2. 操作失败提示统一转译，不展示技术错误字段。
3. `STATUS_CONFLICT` / `INVALID_STATUS` 提示“订单状态已变化，请刷新后重试”。
4. 状态冲突后自动尝试刷新当前列表或详情。
5. 状态更新成功后再刷新列表或详情。
6. 如果状态更新成功但刷新失败，提示“操作成功，但刷新失败，请手动刷新”。
7. 保持 `submittingOrderId` / `submitting` 防重复操作。

## 八、边界说明

1. 当前没有引入支付状态。
2. 当前没有退款流程。
3. 当前取消订单只代表订单状态取消，不代表退款。
4. 后续进入 P3-A 支付前，需要重新设计支付相关状态，例如 `unpaid`、`paid`、`refunding`、`refunded`、`refund_failed`。
5. 当前不做订阅消息。
6. 当前不做实时推送。
7. 当前不接打印机。
8. 当前不做后厨看板。

## 九、风险与注意事项

1. 用户取消订单必须以后端 `openid` 校验为准。
2. 前端隐藏按钮不是安全边界。
3. `updateOrderStatus` 是商家端订单状态操作云函数。
4. `cancelUserOrder` 是用户端取消订单云函数。
5. 商家端和用户端取消职责不要混用。
6. 用户不能取消 `accepted` 之后的订单。
7. 商家不能取消 `cooking` 之后的订单。
8. `finished` / `cancelled` 必须保持终态。
9. 后续接入支付后，取消订单必须重新评估退款影响。
