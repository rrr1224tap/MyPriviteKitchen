# 微信点餐小程序云函数设计文档

版本：V1.0
项目名称：微信点餐小程序
技术方案：微信原生小程序 + 腾讯 CloudBase 云开发
适用阶段：MVP 版本
协作方式：ChatGPT + Codex
文档日期：2026-06-06

---

## 1. 文档目标

本文档用于规范微信点餐小程序的云函数设计，确保 Codex 在后续开发时不会随意创建函数、随意更改入参、绕过权限校验或把敏感逻辑写到小程序前端。

本项目的核心原则是：

1. 前端只负责页面展示和用户操作。
2. 订单、金额、权限、状态流转必须由云函数处理。
3. 用户 openid 必须从云函数上下文获取，不能相信前端传入。
4. 商家权限必须通过 `merchant_staff` 集合校验。
5. 所有订单金额必须由云函数根据数据库餐品价格重新计算。
6. 所有订单状态流转必须在云函数中校验。
7. 所有云函数返回格式必须统一，方便前端处理错误。

所有云函数尽量使用以下错误码。

| 错误码                     | 含义       |
| ----------------------- | -------- |
| SUCCESS                 | 操作成功     |
| INVALID_PARAMS          | 参数错误     |
| UNAUTHORIZED            | 未登录或身份无效 |
| FORBIDDEN               | 无权限      |
| NOT_FOUND               | 数据不存在    |
| VALIDATION_ERROR        | 数据校验失败   |
| ORDER_STATUS_INVALID    | 订单状态不合法  |
| ORDER_STATUS_FLOW_ERROR | 订单状态流转错误 |
| DISH_OFF_SALE           | 餐品已下架    |
| DISH_SOLD_OUT           | 餐品已售罄    |
| STOCK_NOT_ENOUGH        | 库存不足     |
| AMOUNT_ERROR            | 金额计算错误   |
| DATABASE_ERROR          | 数据库操作失败  |
| SERVER_ERROR            | 服务器未知错误  |

## 6. 通用权限规则

### 6.1 获取用户 openid

所有需要识别当前用户身份的云函数，都必须从云函数上下文获取 openid。

禁止使用前端传入的 openid 作为身份依据。

伪代码：

```js
const wxContext = cloud.getWXContext()
const openid = wxContext.OPENID
```

---

### 6.2 校验商家员工身份

商家端所有操作必须校验当前用户是否属于该商家。

需要查询集合：

```text
merchant_staff
```

校验条件：

```js
{
  merchant_id,
  openid,
  status: 'active'
}
```

只有查询到有效员工记录，才能继续执行商家操作。

---

### 6.3 校验订单归属

商家修改订单前，必须校验：

```text
orders.merchant_id === 当前商家 merchant_id
```

普通用户查看订单前，必须校验：

```text
orders.user_openid === 当前用户 openid
```

---

### 6.4 金额安全规则

创建订单时，前端只能传：

```text
dish_id
quantity
remark
pickup_type
contact_phone
```

云函数必须重新读取 `dishes` 集合中的真实价格，并重新计算：

```text
total_amount_cent
subtotal_cent
item_count
```

不能相信前端传入的商品名称、商品价格、订单总金额。

## 7. login 云函数

### 7.1 函数名称

```text
login
```

---

### 7.2 功能说明

用于用户首次进入小程序时完成登录。

主要功能：

1. 获取当前微信用户 openid。
2. 查询 `users` 集合中是否存在该用户。
3. 如果不存在，则创建新用户。
4. 查询该用户是否属于商家员工。
5. 返回用户信息、角色信息、商家信息。

---

### 7.3 入参

```json
{
  "nickname": "微信用户",
  "avatar": "头像地址"
}
```

字段说明：

| 字段       | 类型     | 必填 | 说明   |
| -------- | ------ | -- | ---- |
| nickname | string | 否  | 用户昵称 |
| avatar   | string | 否  | 用户头像 |

---

### 7.4 出参

```json
{
  "success": true,
  "code": "SUCCESS",
  "message": "登录成功",
  "data": {
    "openid": "o_xxxxxx",
    "user": {
      "_id": "user_001",
      "nickname": "微信用户",
      "avatar": "",
      "role": "user",
      "status": "active"
    },
    "merchant_staff": {
      "merchant_id": "merchant_001",
      "role": "owner",
      "status": "active"
    },
    "merchant": {
      "_id": "merchant_001",
      "name": "测试点餐店",
      "business_status": "open"
    }
  }
}
```

如果用户不是商家员工：

```json
{
  "merchant_staff": null,
  "merchant": null
}
```

---

### 7.5 涉及集合

```text
users
merchant_staff
merchants
```

---

### 7.6 业务流程

1. 获取云函数上下文中的 openid。
2. 查询 `users` 集合。
3. 如果用户不存在：

   1. 创建用户。
   2. 默认 `role = user`。
   3. 默认 `status = active`。
4. 如果用户存在：

   1. 更新昵称和头像。
   2. 更新 `updated_at`。
5. 查询 `merchant_staff` 中是否有当前 openid 的 active 记录。
6. 如果有商家员工记录，查询对应 `merchants`。
7. 返回用户信息和商家身份。

---

### 7.7 权限规则

此函数不需要登录前置权限，因为它本身就是登录入口。

---

### 7.8 错误情况

| 情况          | 错误码            |
| ----------- | -------------- |
| 无法获取 openid | UNAUTHORIZED   |
| 数据库查询失败     | DATABASE_ERROR |
| 服务器异常       | SERVER_ERROR   |

### 8.4 出参

```json
{
  "success": true,

    ],

      {

        "category_id": "category_001",
        "name": "招牌牛肉饭",
        "image_url": "",
        "price_cent": 2800,
        "description": "牛肉鲜嫩，搭配米饭和时蔬。",
        "status": "on_sale",
        "stock_enabled": true,
        "stock_count": 99,
        "sold_out": false,
        "sales_count": 0,
        "estimated_time_min": 15,
        "tags": ["招牌", "热卖"]
      }
    ]
  }

```

---

### 8.5 涉及集合

```text
merchants
categories
dishes
```

---

### 8.6 业务流程

1. 校验 `merchant_id` 是否存在。
2. 查询 `merchants` 集合。
3. 如果商家不存在，返回 `NOT_FOUND`。
4. 查询该商家启用的分类：

   1. `merchant_id = 入参 merchant_id`
   2. `enabled = true`
   3. 按 `sort_order` 升序。
5. 查询该商家已上架餐品：

   1. `merchant_id = 入参 merchant_id`
   2. `status = on_sale`
6. 返回餐品库存 / 售罄字段：

   1. `stock_enabled`
   2. `stock_count`
   3. `sold_out`
7. 如果 `on_sale` 餐品已售罄，仍返回给用户端，由用户端显示“已售罄”，而不是隐藏。
8. 返回商家信息、分类、餐品。

---

### 8.7 权限规则

用户端公开菜单可以不要求商家身份。

但是：

1. 只返回已启用分类。
2. 只返回已上架餐品。
3. 已上架但售罄的餐品仍返回，用于用户端展示“已售罄”。
4. 不返回后台字段。
5. 不返回成本、食材内部配置等信息。

---

### 8.8 错误情况

| 情况             | 错误码            |
| -------------- | -------------- |
| merchant_id 缺失 | INVALID_PARAMS |
| 商家不存在          | NOT_FOUND      |
| 数据库查询失败        | DATABASE_ERROR |

## 10. createOrder 云函数

### 10.1 函数名称

```text
createOrder
```

---

### 10.2 功能说明

用户提交订单。

这是本项目最核心的云函数之一，必须保证安全和准确。

主要功能：

1. 获取当前用户 openid。
2. 接收购物车餐品 ID 和数量。
3. 校验餐品是否存在。
4. 校验餐品是否上架。
5. 校验售罄和库存。
6. 根据数据库价格重新计算金额。
7. 创建订单主记录。
8. 创建订单明细记录。
9. 写入新订单提醒。
10. 返回订单 ID 和订单编号。

---

### 10.3 入参

```json
{
  "merchant_id": "merchant_001",
  "items": [
    {
      "dish_id": "dish_001",
      "quantity": 2
    },
    {
      "dish_id": "dish_002",
      "quantity": 1
    }
  ],
  "remark": "不要香菜",
  "pickup_type": "pickup",
  "contact_phone": "13800000000"
}
```

字段说明：

| 字段               | 类型     | 必填 | 说明    |
| ---------------- | ------ | -- | ----- |
| merchant_id      | string | 是  | 商家 ID |
| items            | array  | 是  | 订单餐品  |
| items[].dish_id  | string | 是  | 餐品 ID |
| items[].quantity | number | 是  | 数量    |
| remark           | string | 否  | 用户备注  |
| pickup_type      | string | 是  | 取餐方式  |
| contact_phone    | string | 否  | 联系电话  |

---

### 10.4 出参

```json
{
  "success": true,
  "code": "SUCCESS",
  "message": "订单创建成功",
  "data": {
    "order_id": "order_001",
    "order_no": "20260606153212085",
    "status": "pending",
    "total_amount_cent": 6800,
    "item_count": 3
  }
}
```

---

### 10.5 涉及集合

```text
merchants
dishes
orders
order_items
notifications
operation_logs
```

---

### 10.6 业务流程

1. 获取当前用户 openid。
2. 校验 `merchant_id`。
3. 校验 `items` 不为空。
4. 校验每个 item：

   1. `dish_id` 不能为空。
   2. `quantity` 必须大于 0。
5. 查询商家信息。
6. 校验商家是否存在。
7. 校验商家是否营业：

   1. `business_status = open` 才允许下单。
8. 查询所有涉及的餐品。
9. 对每个餐品进行校验：

   1. 餐品必须存在。
   2. 餐品必须属于当前 `merchant_id`。
   3. 餐品状态必须是 `on_sale`。
   4. 如果 `sold_out === true`，拒绝下单。
   5. 如果 `stock_enabled === true`，则 `stock_count` 必须大于等于购买数量。
10. 使用数据库中的 `price_cent` 重新计算：

    1. 每个商品小计 `subtotal_cent`
    2. 订单总价 `total_amount_cent`
    3. 商品总数量 `item_count`
11. 生成订单编号 `order_no`。
12. 创建 `orders` 记录：

    1. `status = pending`
    2. `payment_status = unpaid`
    3. `payment_method = offline`
13. 创建多条 `order_items` 记录。
14. 如果 `stock_enabled === true`，下单成功后扣减 `stock_count`。
15. 可选：增加餐品销量。
16. 写入 `notifications` 新订单提醒。
17. 写入 `operation_logs`。
18. 返回订单信息。

库存 / 售罄要求：

1. 不信任前端购物车中的库存状态。
2. 每次下单时必须重新查询 `dishes` 集合。
3. 如果 `status !== on_sale`，返回 `DISH_OFF_SALE`。
4. 如果 `sold_out === true`，返回 `DISH_SOLD_OUT`。
5. 如果 `stock_enabled === true` 且 `stock_count < quantity`，返回 `STOCK_NOT_ENOUGH`。
6. MVP 阶段先实现基础扣减；后续需要强化事务或原子扣减，降低并发超卖风险。

---

### 10.7 订单编号生成规则

MVP 阶段推荐：

```text
YYYYMMDDHHmmss + 3位随机数
```

示例：

```text
20260606153212085
```

---

### 10.8 权限规则

1. 必须获取当前用户 openid。
2. 用户只能为自己创建订单。
3. 不能使用前端传入的 openid。
4. 订单金额必须由云函数计算。
5. 餐品名称、价格、图片必须从数据库读取。

---

### 15.6 业务流程

### 15.7 create 逻辑

1. 校验餐品名称不能为空。
2. 校验分类 ID 不能为空。
3. 校验分类属于当前商家。
4. 校验价格 `price_cent > 0`。
5. 校验 `stock_enabled` 必须是 boolean。
6. 校验 `stock_count` 必须是非负整数。
7. 校验 `sold_out` 必须是 boolean。
8. 创建餐品：

   1. 默认 `status = off_sale` 或 `on_sale`。
   2. 默认 `sales_count = 0`。
   3. 写入 `created_at`。
   4. 写入 `updated_at`。
9. 写入操作日志。
10. 返回餐品 ID。

建议 MVP 阶段新增餐品后默认：

```text
off_sale
```

由商家确认后再上架。

---

### 15.8 update 逻辑

1. 校验 `dish_id`。
2. 查询餐品。
3. 校验餐品属于当前商家。
4. 只允许更新安全字段：

   1. `category_id`
   2. `name`
   3. `image_url`
   4. `price_cent`
   5. `description`
   6. `stock_enabled`
   7. `stock_count`
   8. `sold_out`
   9. `estimated_time_min`
   10. `tags`
5. 如果更新分类，必须校验分类属于当前商家。
6. 如果更新库存字段，必须校验：

   1. `stock_enabled` 必须是 boolean。
   2. `stock_count` 必须是非负整数。
   3. `sold_out` 必须是 boolean。
7. 更新 `updated_at`。
8. 写入操作日志。

1. 查询餐品。
2. 校验餐品属于当前商家。
3. 校验餐品名称、价格、分类合法。
4. 建议校验餐品图片不为空。
5. 更新 `status = on_sale`。
6. 更新 `updated_at`。
7. 写入操作日志。

### 15.10 off_sale 逻辑

2. 校验餐品属于当前商家。
3. 更新 `status = off_sale`。

5. 写入操作日志。

3. 取消手动售罄通过 `update` 写入 `sold_out = false`。
4. 如果 `stock_enabled === true` 且 `stock_count <= 0`，也视为售罄。

   1. `stock_enabled`
   2. `stock_count`
   3. `sold_out`

3. 商家餐品管理页增加手动售罄开关。

5. 用户菜单页售罄餐品禁用加购。
6. 用户详情页售罄餐品禁用加入购物车。

```json
            "dish_id": "dish_001",
            "dish_name": "招牌牛肉饭",
            "order_quantity": 2,
            "quantity_per_dish": 120,

          }
```

```text

```

### 17.6 业务流程

1. 获取当前用户 openid。
2. 校验 `merchant_id`。
3. 校验日期范围。
4. 查询 `merchant_staff`，确认当前用户是商家员工。
5. 查询日期范围内订单。
6. 订单状态只统计：

   1. `accepted`
   2. `cooking`
   3. `finished`
7. 不统计：

   1. `pending`
   2. `cancelled`
8. 查询对应 `order_items`。
9. 根据每个 `dish_id` 查询 `dish_ingredients`。
10. 按 `ingredient_id` 汇总数量：

    1. `订单中餐品数量 × 每份餐品所需食材数量`
11. 返回汇总结果。

---

### 17.7 权限规则

只有商家员工可以查看自己商家的食材汇总。

## 21. 前端调用云函数建议

前端统一封装云函数调用方法。

建议创建：

```text
miniprogram/utils/cloud.js
```

统一调用格式：

```js
async function callFunction(name, data = {}) {
  try {
    const res = await wx.cloud.callFunction({
      name,
      data
    })

    const result = res.result

    if (!result.success) {
      wx.showToast({
        title: result.message || '操作失败',
        icon: 'none'
      })
      throw new Error(result.message)
    }

    return result.data
  } catch (error) {
    wx.showToast({
      title: error.message || '网络异常',
      icon: 'none'
    })
    throw error
  }
}
```

前端页面不要直接写大量 `wx.cloud.callFunction`，尽量统一封装。

## 22. Codex 开发要求

Codex 在开发云函数时必须遵守以下规则：

1. 不允许前端直接传 openid 作为身份依据。
2. 不允许前端直接写入订单金额。
3. 不允许前端直接修改订单状态。
4. 不允许普通用户调用商家管理功能成功。
5. 不允许商家跨 merchant_id 操作其他商家数据。
6. 不允许随意新增本文档没有定义的订单状态。
7. 不允许随意新增本文档没有定义的餐品状态。
8. 不允许把支付密钥、商户密钥等敏感信息写进前端。
9. 所有云函数必须使用统一返回格式。
10. 所有云函数必须对参数做基础校验。
11. 所有数据库写操作必须写入 `updated_at`。
12. 关键操作建议写入 `operation_logs`。
13. 新增或修改云函数后，必须更新本文档或对应开发记录。
14. 每次只完成一个云函数或一个小模块，避免一次性大范围修改。

## 26. 当前文档结论

本项目云函数设计的核心是：

1. 用户端通过云函数读取菜单、创建订单、查看订单。
2. 商家端通过云函数查看订单、修改状态、维护餐品和分类。
3. 所有权限、金额、状态流转都必须放在云函数中处理。
4. 前端不能直接信任用户输入，也不能直接修改核心数据库。
5. MVP 阶段先完成登录、菜单、下单、订单管理、餐品管理。
6. 食材汇总和制作流程作为第二批功能继续开发。
7. 微信支付放到 MVP 稳定之后再接入。
