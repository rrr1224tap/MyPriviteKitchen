## 1. 阶段目标

本阶段目标是在进入下一阶段功能开发前，基于当前体验版截图整理一份页面级视觉深度优化清单。

本阶段只做分析和计划：

1. 分析当前体验版各页面的视觉问题。
2. 找出所有明显的“文字 + 圆圈”占位图标。
3. 找出需要替换为真实图片、图标、插画的位置。
4. 梳理首页首屏转化和排版优化方向。
5. 给出资产制作清单和包体积控制原则。

本阶段不做：

1. 不修改 miniprogram 代码。
2. 不修改 cloudfunctions。
3. 不修改 project.config.json。
4. 不替换图片。
5. 不删除图片。
6. 不提交 Git。

## 4. 图片资产设计清单

### 4.1 需要真实菜品图的位置

| 页面 | 位置 | 当前问题 | 建议 |
| --- | --- | --- | --- |
| 首页 | Hero 主视觉 | 当前已有真实拌饭图，方向正确 | 后续换成“小厨食堂”自有招牌菜图，走云存储 |
| 点餐页 | 商品卡片图 | 多个商品仍是棕色渐变占位 | 每个上架餐品至少配 1 张真实菜品图，走云存储 |
| 餐品详情页 | 顶部大图 | 当前复用主视觉，容易和具体餐品不完全对应 | 使用当前餐品真实大图，缺失时显示统一品牌占位 |
| 购物车页 | 商品缩略图 | 当前仍是渐变占位 | 使用商品快照图片或统一缩略图占位 |
| 商家餐品管理 | 餐品缩略图 | 文字“餐”占位明显 | 使用餐品图；无图时统一 placeholder |

### 4.2 可以继续用渐变占位的位置

1. 后台管理中未上传图片的餐品，可以短期保留统一渐变占位。
2. 图片加载失败时，可以保留低调品牌占位图。
3. 草稿 / 下架餐品如果无图，可以使用统一占位，不必强制上真实图。

但用户端点餐主链路不建议继续大量使用渐变占位。

### 4.3 适合插画的位置

1. 空购物车。
2. 空订单列表。
3. 订单详情加载失败。
4. 商家订单列表为空。
5. 商家订单详情加载失败。
6. 菜单加载失败 / fallback 提示。

插画风格建议：

1. 暖色、轻商业、简洁。
2. 不要幼态可爱化。
3. 不要使用复杂大图，控制体积。
4. 插画主体可围绕餐盘、小票、购物车、后厨铃、打包袋。

### 4.4 不应该放进 miniprogram 包的大图

1. 首页大图。
2. 菜品详情大图。
3. 菜品列表真实照片。
4. 促销海报。
5. 设计参考图。

这些图片建议走云存储 URL 或 CDN。

### 4.5 建议走云存储 URL 的图片

1. 所有真实菜品图。
2. 首页主视觉图。
3. 后续“今日推荐”大图。
4. 活动 banner。
5. 商家自定义门店图片。

## 5. 首页排版优化建议

### 5.1 当前优点

1. 顶部品牌主视觉已经足够有餐饮氛围。
2. “小厨食堂”品牌名在首屏可见。
3. “立即点餐”按钮醒目，方向正确。
4. “今天怎么吃”承接堂食 / 自提，用户能理解主流程。

### 5.2 当前问题

1. 首屏 Hero 区高度偏大，底部功能卡片露出有限，小屏用户需要下滑才能完整看到核心入口。
2. 首页下半部分“堂食点餐 / 到店自提 / 我的订单”图标仍是单字圆标，成熟感不足。
3. “今天怎么吃”区域和 Hero 的连接略突兀，缺少一条“推荐 / 门店公告 / 出餐时间”的轻量信息桥。
4. 当前没有“今日推荐”模块，首页除了主视觉外缺少第二个菜品转化点。
5. “商家工作台”入口露在底部，作为用户端首页略显突兀，可以弱化或移动到“我的”入口。

### 5.3 建议修改

P0 建议：

1. 将 Hero 首屏高度略收紧，让“今天怎么吃”的标题和两个入口卡片在首屏更完整露出。
2. 替换堂食、自提、订单、TabBar 的单字图标。
3. 强化“立即点餐”按钮：保留红色主按钮，按钮下可增加轻量提示“预计 15 分钟出餐”。

P1 建议：

1. 增加“今日推荐”轻量模块，建议放在“今天怎么吃”下方或主视觉与入口之间。
2. 增加门店公告 / 出餐时间条，例如：“今日现炒现做，高峰期请耐心等候 · 预计 15 分钟出餐”。
3. “我的订单”入口可以保留，但视觉层级低于点餐入口。

P2 建议：

1. 弱化“商家工作台”入口，避免普通用户误触。
2. 补齐底部服务条图标，例如营业时间、到店自提。

首页目标：

```text
用户 3 秒内看懂：这是什么店、主打什么、怎么点餐、多久能出餐。
```

## 9. 包体积控制原则

1. 不要把真实菜品大图放进小程序包。
2. 不要把首页大图、促销 banner、设计参考图放进小程序包。
3. 本地图标单个尽量小于 20KB。
4. 本地插画建议小于 80KB，超过则考虑压缩或云存储。
5. 菜品图和首页大图建议使用云存储 URL。
6. 设计参考图只能放在 `design_refs` 或被 `packOptions.ignore` 排除的目录。
7. 不要再让 `design_refs` 参与小程序上传包。
8. 图片优先使用 WebP；如兼容性或工具链不稳定，再使用压缩 PNG/JPG。
9. 对于图标，优先统一一套风格，不要混用不同来源。
10. 替换资产后需要重新执行上传包体积检查。

## 12. 当前结论

当前体验版的视觉方向正确，已经具备“暖色商业餐饮风 / 玻璃卡片 / 品牌红按钮”的基础。

深度优化最值得先做的不是大改结构，而是补齐视觉资产：

1. 用真实图标替换单字圆标。
2. 用真实菜品图替换主链路渐变占位。
3. 用统一插画替换空状态 / 错误状态圆点。
4. 调整首页首屏，让点餐入口更快出现。

只要先完成 P0 和 P1，当前体验版的成熟度会明显提升，同时不会影响订单、购物车、规格加料、库存、状态流转等业务逻辑。
