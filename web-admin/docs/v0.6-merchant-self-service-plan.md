# Private-Kitchen-v0.6 邀请码制商户自助开店方案

适用阶段：`Private-Kitchen-v0.6：邀请码制商户自助开店与多商户入口版`

本文用于盘点 v0.5 现有代码基础，并设计“每个朋友都能自己开店”的 MVP 方案。本阶段只做方案设计和代码盘点，不修改业务代码。

## 1. 背景与目标

v0.5 已完成 Web 超级管理员后台基础闭环，包括登录、总览、商户、成员邀请码、分类、餐品、订单、今日备料和发布前总验收文档。

正式上线前，产品目标调整为：先让每个朋友都能通过邀请码自助开店，并拥有自己的 Web 管理入口和顾客小程序入口。

v0.6 的目标：

1. 超级管理员可以发放“开店邀请码”。
2. 朋友使用邀请码创建自己的商户。
3. 朋友成为该商户的 `owner` 或 `merchant_admin`。
4. 朋友能登录 Web 后台，只管理自己的商户。
5. 顾客能通过分享入口进入指定朋友的菜单。
6. 顾客下单后订单绑定到该朋友的商户。
7. 商户能在 Web 后台处理订单并查看今日备料。

## 2. 当前 v0.5 能力基线

Web 后台已完成：

1. `webAdminAuth` 口令登录。
2. `admin_token` 生成与 session 保存。
3. 总览真实读取。
4. 商户列表、新增、编辑、启用 / 禁用。
5. 成员列表、邀请码列表、创建邀请码、禁用邀请码、禁用成员、启用成员。
6. 分类列表、新增、编辑。
7. 餐品列表、新增、编辑、上架 / 下架、`tutorials` 编辑、`ingredients` 编辑。
8. 订单列表、分页、状态筛选、订单详情、状态流转。
9. 今日备料真实读取、日期参数、刷新、空状态、`copy_text` 展示。

小程序端已有：

1. 用户菜单、详情、购物车、下单、订单列表、订单详情。
2. 商家端订单、订单详情、分类、餐品、今日备料。
3. 商家邀请码兑换页：`miniprogram/pages/merchant/invite-redeem/`。

## 3. 当前缺口

1. Web token 目前只支持 `role = super_admin`。
2. Web session 只保存 `token`、`role`、`expires_at`，不保存 `merchant_id`。
3. Web 路由只做“是否有本地 session”的判断，不做角色裁剪。
4. Web 侧分类、餐品、订单、备料页面通过本地 `xiaochu_current_merchant_id` 或 fallback `xiaochu` 选择商户，不能防止商户管理员篡改前端 `merchant_id`。
5. Web `AdminLayout` 固定展示超级管理员菜单和文案。
6. 现有 `merchant_invites` 是“加入某个已存在商户”的邀请码，不是“创建商户”的开店邀请码。
7. 小程序用户端多处仍硬传 `DEFAULT_MERCHANT_ID = merchant_001`。
8. 小程序顾客入口还没有从分享 query 或小程序码解析当前商户。
9. 购物车已有跨商户清空雏形，但当前商户上下文还没有统一存储。
10. 用户订单详情只按 `openid + order_id` 校验，没有显式校验当前入口商户。

## 4. 现有代码盘点

Web 鉴权与请求：

1. `web-admin/src/services/auth.ts`
   - `loginWebAdmin(passcode)` 调用 `webAdminAuth`。
   - 返回类型固定为 `role: 'super_admin'`。
2. `web-admin/src/services/cloud.ts`
   - `callAdminFunction(functionName, payload)` 统一注入 `admin_token`。
   - 只检查本地 session 是否有效。
3. `web-admin/src/stores/session.ts`
   - session 字段为 `token`、`role`、`expires_at`。
   - 暂无 `merchant_id`、`staff_id`、`account_id`。
4. `web-admin/src/router/index.ts`
   - 只根据 `hasValidLocalSession()` 决定是否放行。
   - 不按角色限制页面。
5. `web-admin/src/layouts/AdminLayout.vue`
   - 菜单包含商户管理、成员邀请、分类、餐品、订单、今日备料、数据检查、系统设置。
   - 文案固定为“超级管理员”。
   - 当前商户上下文来自路由参数、本地 `xiaochu_current_merchant_id` 或 fallback `xiaochu`。

Web 页面商户上下文：

1. `CategoriesView.vue` 使用本地 `xiaochu_current_merchant_id` 或 `xiaochu`。
2. `DishesView.vue` 支持从 `/merchants/:merchantId/dishes` 读取路由商户，并写入本地上下文。
3. `OrdersView.vue` 使用本地 `xiaochu_current_merchant_id` 或 `xiaochu`。
4. `PrepSummaryView.vue` 使用本地 `xiaochu_current_merchant_id` 或 `xiaochu`。
5. `MerchantStaffView.vue` 从路由 `:merchantId` 读取商户。

当前仓库中未发现 `web-admin/src/services/admin.ts` 和 `web-admin/src/services/request.ts`，现有统一请求入口是 `web-admin/src/services/cloud.ts`。

已有集合基础：

   - 当前 role 为 `owner` 或 `staff`。
3. `merchant_invites`
   - 关键字段：`code`、`merchant_id`、`role`、`status`、`created_by_openid`、`used_by_openid`、`expires_at`、`created_at`、`used_at`、`updated_at`。
   - 当前 status 包括 `unused`、`used`、`disabled`、`expired`。
4. `categories`
   - 已按 `merchant_id` 隔离。
5. `dishes`
   - 已按 `merchant_id` 隔离。
6. `orders`
   - 已包含 `merchant_id`，Web 订单和备料均按 `merchant_id` 查询。
7. `order_items`
   - 已包含 `merchant_id`，订单详情和备料会按商户过滤。

当前是否具备多商户数据基础：具备基础。核心业务集合已经有 `merchant_id`，分类、餐品、订单、备料在后端大多已经按 `merchant_id` 过滤。但 Web 管理身份还不能把 `merchant_admin` 限定到 token 里的商户，因此权限隔离还没完成。

## 6. 现有邀请码能力判断

现有成员邀请码可以复用的部分：

1. 有随机 `code`。
2. 有 `role` 字段。
3. 有 `expires_at`。
4. 有 `unused / used / disabled / expired` 状态。
5. 有一次性使用逻辑。
6. 有 `used_by_openid` 和 `used_at`。
7. 小程序端已有 `redeemMerchantInvite`，能把 openid 绑定到商户成员。

不适合直接当“开店邀请码”的原因：

1. 现有邀请码必须先绑定到某个 `merchant_id`。
2. 现有语义是“加入某个已存在商户”，不是“创建 / 认领商户”。
3. 现有兑换依赖微信 `OPENID`，不适合直接生成 Web `merchant_admin` 登录凭据。
4. 现有邀请码使用后写入 `merchant_staff`，不创建 `merchants`。
5. 现有 `created_by_openid` 是超级管理员或 Web 管理上下文，不是开店账号体系。

推荐：保留现有 `merchant_invites` 作为“加入商户的邀请码”。开店邀请码优先在不新增集合的方案中复用同集合并新增类型字段；如果后续审计和统计复杂度提高，再评审是否拆出新集合。

## 7. 邀请码制开店推荐方案

推荐 MVP 流程：

1. `super_admin` 在 Web 后台创建“开店邀请码”。
2. 邀请码记录写入 `merchant_invites`，但新增字段标识：
   - `invite_type: 'merchant_signup'`
   - `merchant_id: ''` 或预留为空
   - `role: 'owner'`
   - `status: 'unused'`
3. 朋友打开 Web 开店页，输入邀请码。
4. 朋友填写：
   - 店铺名称 `name`
   - 店铺短名称 `short_name`
   - 登录标识 `merchant_slug`
   - 管理密码
5. 后端校验邀请码未使用、未禁用、未过期。
6. 后端创建 `merchants`。
7. 后端创建 `merchant_staff` owner 记录。
8. 后端把开店邀请码标记为 `used`。
9. 后端签发 `merchant_admin` token。
10. 朋友进入自己的 Web 后台。

推荐先不新增数据库集合：

1. 账号凭据可先放在 `merchant_staff` 或 `merchants` 的受保护字段中。
2. 更推荐放在 `merchant_staff`：
   - `login_name`
   - `password_hash`
   - `password_salt`
   - `account_status`
   - `token_version`
3. 这样 merchant owner 与登录账号归属一致。
4. 后续如果要支持一个人管理多个商户、多账号、多手机号，再拆 `merchant_accounts` 集合。

需开发时再确认：

1. `merchant_slug` 是否等同 `merchant_id`。
2. 密码规则和找回密码策略。
3. 是否要求绑定微信 openid。

## 8. Web merchant_admin 登录方案

方案 A：商户码 / 手机号 / 密码登录

优点：

1. 用户理解成本低。
2. 未来适合接短信、手机号找回。

缺点：

1. 当前项目没有手机号验证和短信能力。
2. 需要新增手机号唯一性和找回密码设计。
3. 容易在 MVP 阶段引入额外合规和安全成本。

方案 B：邀请码首次开店后设置密码，再用 `merchant_slug + password` 登录

优点：

1. 最贴合当前“邀请码制开店”目标。
2. 不依赖微信 Web 登录、不依赖短信。
3. 不新增外部服务。
4. `merchant_slug` 可以直接用于分享入口和 Web 登录标识。
5. 易于和 `merchant_id` 做一一映射。

缺点：

1. 需要妥善处理密码哈希、重置密码和重复 slug。
2. 后续多账号能力需要扩展。

方案 C：依赖微信 openid 绑定后登录

优点：

1. 和小程序商家身份体系一致。
2. 不需要 Web 密码。

缺点：

1. Web 端直接拿微信 openid 成本高。
2. 需要微信 OAuth 或扫码登录流程，超出当前 MVP。
3. 当前 Web 后台是浏览器后台，不是小程序内后台。

推荐 MVP：方案 B。先做“开店邀请码 + merchant_slug + password”。后续再补手机号或微信绑定。

## 9. Web Token Payload 设计

当前 token payload：

1. `role`
2. `issued_at`
3. `expires_at`
4. `nonce`

v0.6 推荐 payload：

```json
{
  "role": "merchant_admin",
  "merchant_id": "friend_kitchen",
  "staff_id": "staff_xxx",
  "account_id": "staff_xxx",
  "issued_at": "2026-07-04T00:00:00.000Z",
  "expires_at": "2026-07-04T04:00:00.000Z",
  "token_version": 1,
  "nonce": "random"
}
```

`super_admin` token 可继续保持：

```json
{
  "role": "super_admin",
  "issued_at": "...",
  "expires_at": "...",
  "nonce": "random"
}
```

设计要求：

1. `role = super_admin` 可以指定 `merchant_id`。
2. `role = merchant_admin` 必须带 `merchant_id`。
3. 后端对 `merchant_admin` 忽略前端传入的跨商户 `merchant_id`，统一使用 token 内的 `merchant_id`。
4. `token_version` 用于未来强制退出登录，MVP 可先预留。

## 11. 云函数权限升级方案

核心原则：

1. `super_admin`：允许指定 `merchant_id`。
2. `merchant_admin`：只能使用 token 内的 `merchant_id`。
3. 任何写操作都不能信任前端传入的商户 ID。
4. 所有 Web 相关云函数都需要返回统一友好错误。

需要升级 token helper 的云函数：

1. `getAdminOverview`
2. `manageMerchant`
3. `manageMerchantStaff`
4. `manageCategory`
5. `manageDish`
6. `getMerchantOrders`
7. `getMerchantOrderDetail`
8. `updateOrderStatus`
9. `getPrepSummary`

具体升级方式：

1. 统一 `verifyWebAdminToken` 返回 payload，而不只是 role。
2. 增加 `resolveWebMerchantId(payload, eventMerchantId)`：
   - `super_admin` 返回前端传入的 `merchant_id`。
   - `merchant_admin` 返回 token 中的 `merchant_id`。
   - 如果 `merchant_admin` 前端传了不同商户，返回 `FORBIDDEN` 或直接忽略并记录风险日志，推荐返回 `FORBIDDEN`。
3. `manageMerchant`
   - `super_admin` 保留列表、新增、编辑、启停。
   - `merchant_admin` 仅允许读取或更新自己商户基础资料，MVP 可先不开放。
4. `manageMerchantStaff`
   - `super_admin` 保留当前能力。
   - `merchant_admin` MVP 暂不开放成员管理。
5. `manageCategory`
   - `merchant_admin` 允许 list/create/update 自己商户分类。
6. `manageDish`
   - `merchant_admin` 允许 list/create/update/status/tutorials/ingredients 自己商户餐品。
7. `getMerchantOrders`
   - `merchant_admin` 只能读取自己商户订单。
8. `getMerchantOrderDetail`
   - `merchant_admin` 只能读取自己商户订单详情。
9. `updateOrderStatus`
   - `merchant_admin` 只能更新自己商户订单，状态流转保持 `pending -> accepted -> cooking -> finished`。
10. `getPrepSummary`
    - `merchant_admin` 只能读取自己商户备料。

最容易越权的位置：

1. Web 页面传入的 `merchant_id` 被后端直接信任。
2. `localStorage` 中的 `xiaochu_current_merchant_id` 可被用户篡改。
3. 订单详情和状态更新如果只按 `order_id` 查找，必须再校验 `order.merchant_id`。
4. 分类和餐品新增时，必须使用 token 解析后的商户 ID 写入。
5. 今日备料必须用 token 商户过滤 `orders`、`order_items`、`dishes`。

## 12. 小程序多商户入口方案

推荐入口参数：

1. `merchant_id`：直接使用商户 ID。
2. `merchant_slug`：更适合分享展示，后端再解析为 `merchant_id`。

MVP 推荐：先用 `merchant_id`，后续再补 `merchant_slug`。

分享入口示例：

```text
/pages/user/menu/menu?merchant_id=friend_kitchen
```

小程序码参数示例：

```text
merchant_id=friend_kitchen
```

小程序需要新增当前商户上下文工具：

1. 读取页面 query 的 `merchant_id`。
2. 校验为空时 fallback `DEFAULT_MERCHANT_ID`。
3. 存储到本地，例如 `current_merchant_id`。
4. 切换商户时检查购物车商户。
5. 如果购物车不是当前商户，提示并清空或要求用户确认。

需要修改的小程序文件：

1. `miniprogram/utils/constants.js`
2. `miniprogram/utils/cart.js`
3. 建议新增 `miniprogram/utils/merchant-context.js`
4. `miniprogram/pages/user/menu/menu.js`
5. `miniprogram/pages/user/dish-detail/dish-detail.js`
6. `miniprogram/pages/user/cart/cart.js`
7. `miniprogram/pages/user/order-list/order-list.js`
8. `miniprogram/pages/user/order-detail/order-detail.js`
9. `miniprogram/pages/user/submit-order/submit-order.js`
10. 商家端页面后续也应从当前员工身份或选择商户读取，不再硬编码 `DEFAULT_MERCHANT_ID`。

无效或禁用商户处理：

1. `getMenu` 返回 `NOT_FOUND` 或 `MERCHANT_DISABLED`。
2. 页面显示“店铺不存在或暂不可用”。
3. 提供回到默认店铺或重新扫码入口。

## 13. 购物车与订单 merchant_id 方案

购物车：

1. 当前 `cart.js` 已保存 `merchant_id`。
2. 当前 `addCartItem` 已在发现不同商户时清空旧购物车。
3. v0.6 需要把当前商户上下文统一传给菜单、详情和购物车。
4. 切换商户时建议弹窗确认：“切换店铺会清空当前购物车”。

下单：

1. `createOrder` 必须使用当前商户上下文的 `merchant_id`。
2. 前端只传 `merchant_id` 和 `items` 的 `dish_id / quantity / selected_specs / selected_addons`。
3. 后端继续重新查询商户和餐品，不信任前端价格。
4. 后端必须确认餐品属于当前 `merchant_id`。
5. 订单和订单明细都写入当前 `merchant_id`。

订单列表：

1. 用户端订单列表可按当前商户展示，也可展示用户全部订单。
2. MVP 推荐：菜单入口内默认按当前 `merchant_id` 筛选。
3. 后续可增加“全部店铺订单”视图。

订单详情：

1. 当前 `getOrderDetail` 按 `openid + order_id` 校验。
2. v0.6 推荐可选传 `merchant_id`。
3. 如果传入 `merchant_id`，后端应校验订单属于该商户。
4. 这能避免用户在某个店铺入口打开另一店铺订单造成体验混乱。

## 14. 安全边界和越权风险

高风险点：

1. `merchant_admin` 不能通过修改请求体 `merchant_id` 访问其他商户。
2. Web 本地 session、localStorage、路由参数都不可信。
3. 开店邀请码必须一次性使用，且创建商户和标记邀请码使用最好在同一事务或补偿逻辑中处理。
4. 密码必须只保存哈希和盐，不保存明文。
5. `merchant_slug` / `merchant_id` 必须唯一且不可随意修改。
6. 禁用商户后，Web 登录、菜单读取、下单都应拒绝或提示不可用。
7. 老的 `DEFAULT_MERCHANT_ID` fallback 不能掩盖无效入口，否则顾客可能进入错店。

## 15. 需要新增 / 复用的字段

优先不新增集合，建议复用 `merchants`、`merchant_staff`、`merchant_invites`。

`merchant_invites` 新增字段建议：

1. `invite_type`: `'staff_join' | 'merchant_signup'`
2. `signup_name`: 可选，记录开店申请人名称，MVP 可不做。
3. `used_merchant_id`: 开店成功后记录创建出的商户。

`merchant_staff` 新增字段建议：

1. `login_name`
2. `password_hash`
3. `password_salt`
4. `account_status`: `'active' | 'disabled'`
5. `token_version`
6. `last_login_at`

`merchants` 新增字段建议：

1. `merchant_slug`
2. `business_status` 可继续沿用现有菜单 / 下单逻辑。
3. `created_from_invite_code`

是否需要新增数据库集合：

1. MVP 推荐不新增集合。
2. 如果后续需要一个账号管理多个商户、手机号登录、密码重置、登录审计，再新增 `merchant_accounts` 或 `merchant_login_accounts`。
3. 当前阶段如新增集合，会增加权限迁移和测试成本，不建议作为第一步。

## 18. 测试计划

后端测试：

1. `webAdminAuth`：
   - `super_admin` 登录仍可用。
   - `merchant_admin` 登录成功。
   - token payload 包含 `merchant_id`。
   - token 篡改、过期、role 非法失败。
2. 开店云函数：
   - 有效开店邀请码成功创建商户。
   - 邀请码为空、错误、已用、禁用、过期失败。
   - 重复 `merchant_id` / `merchant_slug` 失败。
   - 密码不会明文保存。
   - 成功后创建 owner staff。
3. 权限隔离：
   - `merchant_admin` 访问自己商户成功。
   - `merchant_admin` 请求其他商户失败。
   - `super_admin` 指定商户仍成功。
4. 小程序：
   - `getMenu` 使用指定 `merchant_id`。
   - `createOrder` 绑定指定 `merchant_id`。
   - `getUserOrders` 按商户筛选。
   - `getOrderDetail` 可校验商户上下文。

Web 构建：

1. 每个阶段执行 `cd web-admin && npm run build`。

## 19. 手动验收计划

完整验收链路：

1. `super_admin` 登录 Web。
2. `super_admin` 创建开店邀请码。
3. 朋友打开开店页并输入邀请码。
4. 朋友填写店铺名称、短名称、slug、密码。
5. 系统创建商户和 owner。
6. 朋友登录 Web。
7. 朋友只能看到自己的后台菜单。
8. 朋友创建分类。
9. 朋友创建餐品。
10. 朋友配置食材。
11. 朋友分享自己的小程序入口。
12. 顾客进入朋友菜单。
13. 顾客下单。
14. 朋友在 Web 看到订单。
15. 朋友更新订单状态。
16. 朋友查看今日备料。
17. 尝试篡改 `merchant_id` 请求其他商户，确认失败。

## 20. 风险点

1. 最大风险：`merchant_admin` 越权访问其他商户数据。
2. 开店邀请码使用过程如果部分成功部分失败，可能产生半创建商户。
3. 现有小程序默认商户 fallback 可能导致顾客进入错店。
4. 密码体系需要避免明文存储和弱密码。
5. Web 当前布局、文案、菜单均偏 `super_admin`，需要系统性裁剪。
6. 现有成员邀请码与开店邀请码若混用，需要明确 `invite_type`，否则语义混乱。

## 21. 暂不做能力

v0.6 MVP 暂不做：

1. 微信支付。
2. 退款。
3. 打印机 / 后厨出单。
4. 取消订单。
5. 删除订单。
6. 库存管理。
7. 采购单编辑。
8. 手机短信登录。
9. 微信扫码 Web 登录。
10. 一个账号管理多个商户。
11. 普通员工 Web 分级权限。
12. 规格 / 加料 Web 编辑。
13. 商户入驻审核流。
