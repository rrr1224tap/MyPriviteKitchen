# Private-Kitchen-v0.6 开店邀请码数据模型与接口设计

适用阶段：`Private-Kitchen-v0.6：邀请码制商户自助开店与多商户入口版`

本文用于承接 `v0.6-merchant-self-service-plan.md`，进一步明确“开店邀请码”的数据模型、接口边界、权限规则、前端入口和测试清单。本阶段只做设计文档，不修改业务源码。

## 1. 背景与目标

v0.5 已完成 Web 超级管理员后台、商户管理、成员邀请、分类餐品、订单和今日备料的真实接入。v0.6 的核心变化是：朋友拿到开店邀请码后，可以自己创建商户，并获得自己的 Web 后台管理入口。

本设计要解决的问题：

1. 开店邀请码是否复用现有 `merchant_invites`。
2. 开店后商户、店主账号和邀请码如何关联。
3. Web 后台如何区分 `super_admin` 和 `merchant_admin`。
4. 商户管理员如何登录。
5. 如何避免“邀请码已使用但商户没创建成功”这类半成功状态。

本设计不做代码实现，不提交 Git，不创建 tag。

## 2. 与 A1 方案的关系

A1 已确认 v0.6 不再继续 v0.5 稳定 tag，而是先补齐“每个朋友都能自己开店”的能力。当前代码基础已经具备多商户数据底座：

1. `merchants` 保存商户主数据。
2. `merchant_staff` 保存商户成员。
3. `merchant_invites` 保存现有成员邀请码。
4. `categories`、`dishes`、`orders`、`order_items`、今日备料均已围绕 `merchant_id` 做业务隔离。

A2 在这个基础上建议：**不新增数据库集合，复用现有集合，并通过字段扩展区分“成员加入邀请码”和“开店邀请码”。**

## 3. 最终推荐方案

推荐方案：

1. 复用 `merchant_invites`，新增 `invite_type` 区分邀请码类型。
2. `invite_type = staff_join` 表示加入已有商户。
3. `invite_type = merchant_signup` 表示创建新商户。
4. 商户账号信息优先存放在 `merchant_staff` 的 owner 记录中。
5. Web 商户管理员 token 新增 `merchant_admin` 角色。
6. 推荐新增独立云函数 `merchantSelfService` 承载开店、预览邀请码和商户登录接口。
7. 本设计阶段不新增云函数，后续实现阶段再创建。

不推荐新建数据库集合的原因：

1. MVP 阶段复用现有集合即可表达开店邀请、商户和 owner 关系。
2. 现有核心业务集合已按 `merchant_id` 隔离。
3. 新集合会增加迁移、权限和测试复杂度。

后续如果出现一个账号管理多个商户、手机号登录、找回密码、审计统计等复杂场景，再评估新增 `merchant_accounts` 或独立 invite 集合。

## 4. 数据模型设计

### 4.1 merchant_invites

`merchant_invites` 继续作为邀请码集合，新增类型字段后同时支持两类场景。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `code` | string | 是 | 邀请码，唯一 |
| `invite_type` | string | 是 | `staff_join` 或 `merchant_signup` |
| `merchant_id` | string | 否 | 成员邀请码绑定已有商户；开店邀请码可为空 |
| `used_merchant_id` | string | 否 | 开店成功后写入新商户 ID |
| `role` | string | 是 | 成员邀请码可为 `owner` / `staff`；开店邀请码固定为 `owner` |
| `status` | string | 是 | `unused` / `used` / `disabled` / `expired` |
| `remark` | string | 否 | 管理员备注 |
| `created_by_role` | string | 是 | 建议写入 `super_admin` |
| `created_by_openid` | string | 否 | 小程序侧创建者 openid，Web 创建可为空 |
| `created_by_account_id` | string | 否 | Web 创建者账号 ID，当前 super_admin 可先为空 |
| `used_by_openid` | string | 否 | 微信侧使用者 openid，Web 开店可为空 |
| `used_by_account_id` | string | 否 | 开店后 owner staff/account 标识 |
| `expires_at` | Date/string | 是 | 过期时间 |
| `created_at` | Date/string | 是 | 创建时间 |
| `updated_at` | Date/string | 是 | 更新时间 |
| `used_at` | Date/string | 否 | 使用时间 |
| `disabled_at` | Date/string | 否 | 禁用时间 |

兼容规则：

1. 历史成员邀请码如果没有 `invite_type`，后端应按 `staff_join` 处理。
2. `staff_join` 必须有 `merchant_id`。
3. `merchant_signup` 创建时 `merchant_id` 可以为空，兑换成功后写入 `used_merchant_id`。
4. 现有 `unused / used / disabled / expired` 状态继续复用，不新增状态。

### 4.2 merchants

`merchants` 保存开店后生成的商户。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `merchant_id` | string | 是 | 内部商户 ID，可由 slug 规范化生成或使用 `merchant_${shortid}` |
| `merchant_slug` | string | 是 | 外部登录、分享和识别用的短标识 |
| `name` | string | 是 | 商户名称 |
| `short_name` | string | 是 | 商户短名称 |
| `status` | string | 是 | `active` / `disabled` |
| `owner_openid` | string | 否 | 如果后续绑定微信，可写入 |
| `owner_staff_id` | string | 是 | owner 对应的 `merchant_staff.staff_id` |
| `created_from_invite_code` | string | 是 | 来源开店邀请码 |
| `notice` | string | 否 | 商户公告 |
| `created_at` | Date/string | 是 | 创建时间 |
| `updated_at` | Date/string | 是 | 更新时间 |

`merchant_slug` 与 `merchant_id` 建议：

1. `merchant_slug` 是给人看的入口标识，例如 `xiaochu-demo`。
2. `merchant_id` 是内部数据隔离 ID。
3. MVP 可先让 `merchant_id = merchant_slug` 的规范化结果，便于排查。
4. 如果担心后续修改 slug 影响数据关系，可用 `merchant_${shortid}` 作为 `merchant_id`，并保持 `merchant_slug` 唯一。
5. 无论采用哪种方式，后端必须做唯一性检查。

### 4.3 merchant_staff

`merchant_staff` 保存成员，同时保存 owner 的 Web 登录账号。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `staff_id` | string | 是 | 员工记录 ID |
| `merchant_id` | string | 是 | 所属商户 |
| `openid` | string | 否 | 绑定微信后写入；纯 Web 开店可为空 |
| `role` | string | 是 | `owner` / `staff` |
| `status` | string | 是 | 成员状态，`active` / `disabled` |
| `nickname` | string | 否 | 昵称 |
| `remark` | string | 否 | 备注 |
| `login_name` | string | 是 | Web 商户管理员登录名 |
| `password_hash` | string | 是 | 密码哈希，不保存明文 |
| `password_salt` | string | 是 | 密码盐 |
| `account_status` | string | 是 | `active` / `disabled` |
| `token_version` | number | 是 | token 失效控制，默认 1 |
| `last_login_at` | Date/string | 否 | 最近登录时间 |
| `created_at` | Date/string | 是 | 创建时间 |
| `updated_at` | Date/string | 是 | 更新时间 |

账号存放在 `merchant_staff` 的原因：

1. owner 本身就是商户成员。
2. 权限、状态和商户归属可以放在同一条记录里判断。
3. 暂不引入 `merchant_accounts`，降低 MVP 复杂度。

## 5. 密码与登录安全设计

密码规则：

1. 不允许保存明文密码。
2. 不新增 npm 依赖。
3. 使用 Node.js 内置 `crypto`。
4. 推荐使用 `crypto.scrypt`；如果实现环境不方便，可使用 `crypto.pbkdf2`。
5. 每个账号单独生成 `password_salt`。
6. `password_hash` 只保存派生后的哈希结果。
7. 密码最小长度建议 8 位。
8. 登录失败统一返回 `INVALID_LOGIN`，不提示“商户不存在”或“密码错误”。

登录时必须同时校验：

1. `merchant_slug` 存在。
2. 商户 `status = active`。
3. owner/staff 账号存在。
4. `merchant_staff.status = active`。
5. `account_status = active`。
6. 密码哈希匹配。

## 6. merchant_admin token 设计

现有 `webAdminAuth` token 只支持 `role = super_admin`。v0.6 需要支持商户管理员 token。

推荐 `merchant_admin` token payload：

```json
{
  "role": "merchant_admin",
  "merchant_id": "merchant_001",
  "staff_id": "staff_xxx",
  "account_id": "staff_xxx",
  "login_name": "owner",
  "issued_at": 1783120000000,
  "expires_at": 1783206400000,
  "token_version": 1,
  "nonce": "random"
}
```

权限规则：

1. `super_admin` 可以管理全部商户和创建开店邀请码。
2. `merchant_admin` 只能管理 token 中的 `merchant_id`。
3. Web 前端传入的 `merchant_id` 不能作为最终权限依据。
4. 后端收到 `merchant_admin` token 后，应优先使用 token 中的 `merchant_id`。
5. 如果请求体中的 `merchant_id` 与 token 中不一致，应拒绝或强制改为 token 中的商户，推荐拒绝并返回 `FORBIDDEN`。
6. 商户被禁用后，商户管理员不能登录，也不能继续使用旧 token 操作。
7. `token_version` 变化后，旧 token 应失效。

## 7. 云函数放置方案比较

### 方案 A：复用 webAdminAuth

优点：

1. 已有 Web token 签发逻辑。
2. 登录语义接近。

缺点：

1. `webAdminAuth` 当前语义是超级管理员口令登录。
2. 混入开店兑换和商户账号登录后职责会变重。
3. 容易把 `super_admin` 和 `merchant_admin` 的安全边界写混。

### 方案 B：复用 manageMerchantStaff

优点：

1. 已有 `merchant_invites` 创建和禁用逻辑。
2. 成员邀请码与开店邀请码都属于 invite 领域。

缺点：

1. `manageMerchantStaff` 主要语义是管理已有商户成员。
2. 开店邀请码兑换会创建 `merchants` 和 owner，不只是 staff。
3. 未登录用户兑换开店邀请码放在 staff 管理函数里语义不清。

### 方案 C：新增 merchantSelfService

优点：

1. 清晰承载开店邀请码、开店兑换、商户账号登录。
2. 可以把匿名开店接口与超级管理员接口明确分开。
3. 后续扩展找回密码、开店审核、绑定微信时边界更清楚。

缺点：

1. 需要新增云函数。
2. 需要新增 HTTP 网关配置。
3. 需要补充完整测试。

最终推荐：**方案 C，新增 `cloudfunctions/merchantSelfService`。**

说明：这只是 A2 设计建议，不在本阶段创建云函数；新增云函数也不等于新增数据库集合。

## 8. 推荐接口设计

### 8.1 createMerchantSignupInvite

用途：超级管理员创建开店邀请码。

权限：仅 `super_admin`。

请求：

```json
{
  "action": "createMerchantSignupInvite",
  "admin_token": "super admin token",
  "expires_in_days": 7,
  "remark": "给朋友 A 的开店邀请码"
}
```

写入：

1. 只写入 `merchant_invites`。
2. `invite_type = merchant_signup`。
3. `role = owner`。
4. `status = unused`。

响应：

```json
{
  "success": true,
  "code": "SUCCESS",
  "message": "开店邀请码已创建",
  "data": {
    "invite": {
      "code": "ABCDEFGH",
      "invite_type": "merchant_signup",
      "status": "unused",
      "expires_at": "2026-07-11T00:00:00.000Z",
      "created_at": "2026-07-04T00:00:00.000Z"
    }
  }
}
```

### 8.2 previewMerchantSignupInvite

用途：开店页预览邀请码是否可用。

权限：无需登录。

请求：

```json
{
  "action": "previewMerchantSignupInvite",
  "code": "ABCDEFGH"
}
```

响应只返回非敏感信息：

```json
{
  "success": true,
  "code": "SUCCESS",
  "message": "邀请码可用",
  "data": {
    "invite": {
      "invite_type": "merchant_signup",
      "status": "unused",
      "expires_at": "2026-07-11T00:00:00.000Z"
    }
  }
}
```

### 8.3 redeemMerchantSignupInvite

用途：朋友使用开店邀请码创建商户和 owner 账号。

权限：无需登录，但必须持有有效开店邀请码。

请求：

```json
{
  "action": "redeemMerchantSignupInvite",
  "code": "ABCDEFGH",
  "merchant_name": "小厨食堂朋友店",
  "short_name": "朋友店",
  "merchant_slug": "friend-kitchen",
  "login_name": "owner",
  "password": "password input"
}
```

成功后写入：

1. 新增 `merchants`。
2. 新增 `merchant_staff` owner。
3. 更新 `merchant_invites.status = used`。
4. 更新 `merchant_invites.used_merchant_id`。
5. 更新 `merchant_invites.used_by_account_id`。
6. 更新 `merchant_invites.used_at` 和 `updated_at`。

响应：

```json
{
  "success": true,
  "code": "SUCCESS",
  "message": "开店成功",
  "data": {
    "merchant": {
      "merchant_id": "friend-kitchen",
      "merchant_slug": "friend-kitchen",
      "name": "小厨食堂朋友店",
      "short_name": "朋友店",
      "status": "active"
    },
    "session": {
      "token": "merchant admin token",
      "role": "merchant_admin",
      "merchant_id": "friend-kitchen",
      "expires_at": 1783206400000
    }
  }
}
```

### 8.4 merchantAdminLogin

用途：商户管理员后续登录 Web 后台。

权限：无需登录。

请求：

```json
{
  "action": "merchantAdminLogin",
  "merchant_slug": "friend-kitchen",

  "password": "password input"
}
```

响应：

```json
  "code": "SUCCESS",
  "message": "登录成功",
  "data": {
    "session": {
      "token": "merchant admin token",
      "role": "merchant_admin",
      "merchant_id": "friend-kitchen",
      "expires_at": 1783206400000
    },
    "merchant": {
      "merchant_id": "friend-kitchen",
      "merchant_slug": "friend-kitchen",

```

## 9. 错误码设计

| 错误码 | 说明 |
| --- | --- |
| `INVALID_ACTION` | 操作类型不合法 |
| `UNAUTHORIZED` | 未登录或 token 无效 |
| `FORBIDDEN` | 没有权限 |
| `INVALID_INVITE_CODE` | 邀请码格式不合法 |
| `INVITE_NOT_FOUND` | 邀请码不存在 |
| `INVITE_TYPE_MISMATCH` | 邀请码类型不匹配 |
| `INVITE_USED` | 邀请码已使用 |
| `INVITE_DISABLED` | 邀请码已禁用 |
| `INVITE_EXPIRED` | 邀请码已过期 |
| `MERCHANT_SLUG_REQUIRED` | 商户标识必填 |
| `MERCHANT_SLUG_INVALID` | 商户标识格式不合法 |
| `MERCHANT_SLUG_EXISTS` | 商户标识已存在 |
| `MERCHANT_NAME_REQUIRED` | 商户名称必填 |
| `LOGIN_NAME_REQUIRED` | 登录名必填 |
| `LOGIN_NAME_INVALID` | 登录名格式不合法 |
| `PASSWORD_REQUIRED` | 密码必填 |
| `PASSWORD_TOO_SHORT` | 密码太短 |
| `PASSWORD_WEAK` | 密码强度不足 |
| `MERCHANT_CREATE_FAILED` | 商户创建失败 |
| `OWNER_CREATE_FAILED` | 店主账号创建失败 |
| `INVITE_MARK_USED_FAILED` | 邀请码标记使用失败 |
| `MERCHANT_DISABLED` | 商户已禁用 |
| `ACCOUNT_DISABLED` | 账号已禁用 |
| `INVALID_LOGIN` | 登录信息不正确 |
| `TOKEN_INVALID` | token 无效 |
| `TOKEN_EXPIRED` | token 已过期 |
| `INTERNAL_ERROR` | 系统内部错误 |

错误提示原则：

1. 面向用户的提示要友好。
2. 登录失败统一用 `INVALID_LOGIN`，避免泄露商户或账号是否存在。
3. 响应中不得返回 `password_hash`、`password_salt`、token secret 或内部堆栈。
4. 前端可以根据错误码展示更清楚的文案。

## 10. 避免半成功状态

开店兑换必须按以下顺序处理：

1. 校验邀请码存在。
2. 校验 `invite_type = merchant_signup`。
3. 校验状态为 `unused`。
4. 校验未过期。
5. 校验 `merchant_slug` 唯一。
6. 校验表单字段。
7. 创建 `merchants`。
8. 创建 `merchant_staff` owner。
9. 标记邀请码为 `used`。
10. 签发 `merchant_admin` token。

优先使用数据库事务。如果当前 CloudBase 环境事务支持受限，则必须设计补偿和幂等逻辑：

1. 如果商户创建成功但 owner 创建失败，应删除刚创建的商户或标记为不可用，并返回失败。
2. 如果 owner 创建成功但邀请码标记失败，应回滚 owner 和商户，或允许使用同一 `code + merchant_slug` 重试并补齐状态。
3. 不允许出现“邀请码已 used，但没有商户”的状态。
4. 不允许出现“商户存在，但没有 owner”的状态。
5. 重试时如果发现 `created_from_invite_code` 已对应完整商户和 owner，可以返回成功并重新签发 token。

## 11. 前端页面设计

### 11.1 开店页

建议新增 Web 页面：`/merchant-signup`

页面特点：

1. 不需要登录。
2. 可以从 URL 读取 `code` 参数。
3. 先调用 `previewMerchantSignupInvite` 判断邀请码是否可用。
4. 表单字段：商户名称、商户短名称、商户标识、登录名、密码。
5. 提交调用 `redeemMerchantSignupInvite`。
6. 成功后保存 `merchant_admin` session。
7. 自动进入该商户后台。

### 11.2 登录页

现有 LoginView 保留超级管理员口令登录。

建议新增商户管理员登录入口：

1. `super_admin`：继续使用口令。
2. `merchant_admin`：使用 `merchant_slug + login_name + password`。

### 11.3 后台菜单

`super_admin` 菜单：

1. 概览。
2. 商户管理。
3. 成员邀请。
4. 分类管理。
5. 餐品管理。
6. 订单管理。
7. 今日备料。
8. 数据检查。
9. 系统设置。

`merchant_admin` 菜单：

1. 概览。
2. 分类管理。
3. 餐品管理。
4. 订单管理。
5. 今日备料。

`merchant_admin` 不应看到商户管理和跨商户成员邀请入口。

## 12. 后端权限边界

已有云函数后续需要支持两种 Web token：

1. `super_admin`：按现有逻辑可跨商户管理。
2. `merchant_admin`：只能访问 token 中的 `merchant_id`。

需要重点改造的函数：

1. `manageCategory`
2. `manageDish`
3. `getMerchantOrders`
4. `getMerchantOrderDetail`
5. `updateOrderStatus`
6. `getPrepSummary`

处理规则：

1. 如果 token 是 `merchant_admin`，后端不能相信前端传入的 `merchant_id`。
2. 写操作必须只落到 token 中的商户。
3. 商户 `status = disabled` 时拒绝管理操作。
4. 成员账号 `status` 或 `account_status` 禁用时拒绝操作。

## 16. 最大风险

前三个最大风险：

1. **权限隔离风险**：如果后端继续相信前端传入的 `merchant_id`，商户管理员可能越权访问其他商户数据。
2. **开店半成功风险**：商户、owner、邀请码三步写入如果没有事务或补偿，可能留下脏数据。
3. **账号安全风险**：密码哈希、登录错误提示、token role 判断如果处理不严，会影响所有商户后台安全。
